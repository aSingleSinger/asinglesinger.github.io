1. 简介

1.1 介绍

Spring Web MVC 是基于 Servlet API 构建的原始 Web 框架。

==原始 Servlet API 开发代码片段==

// 实现Servlet接口
public class HelloServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException {
        //取得参数
        String method = req.getParameter("method");

        //业务逻辑
        //视图跳转
        req.getRequestDispatcher("/WEB-INF/jsp/hello.jsp").forward(req,resp);
    }
 
    @Override
    protected void doPost(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException {
        doGet(req,resp);
    }
}

在 web.xml 中注册 Servlet

<!-- 为 Servlet 配置url -->
<servlet>
    <servlet-name>HelloServlet</servlet-name>
    <servlet-class>com.lh.servlet.HelloServlet</servlet-class>
</servlet>
<servlet-mapping>
    <servlet-name>HelloServlet</servlet-name>
    <url-pattern>/user</url-pattern>
</servlet-mapping>

==基于 SpringMVC 开发代码片段==

@RequestMapping("/user/login") // url路径
                                              // 快速获取参数
public String login(@RequestParam("userName") String userName, String password){
    
    log.debug("userName" + userName);
    // 调用业务即可
    return "result";
}

SpringMVC 相对于原来的 Servlet API 的优点

  1. 简化接收前端参数 (不需要 request.getParameter(参数名))
  2. 简化后端数据响应 (不需要 reponse.getRequestDispatcher("/WEB-INF/jsp/test.jsp").forward(req,resp))
  3. url 映射使用更简单的 @RequestMapping 进行替换

1.2 内部流程和核心组件

  1. DispatcherServlet: SpringMVC 提供, 我们需要使用 web.xml 配置使其生效, 它是整个流程处理的核心, 所有请求都经过它的处理和分发
  2. HandlerMapping: SpringMVC 提供, 我们需要进行 IOC 配置使其加入 IOC 容器方可生效, 它内部缓存 handler(controller方法)和 handler 访问路径数据, 被 DispatcherServlet 调用, 用于查找路径对应的 handler
  3. HandlerAdapter: SpringMVC 提供, 我们需要进行 IOC 配置使其加入 IOC 容器方可生效, 它可以处理请求参数和处理响应数据 (DispatcherServlet 给 HandlerAdapter 的都是 request, 需要做一些处理, 使 handler 通过参数列表接收数据), 每次 DispatcherServlet 都是通过 handlerAdapter 间接调用 handler, 它是它们之间的适配器。
  4. Handler: handler 又称处理器, 它是 Controller 类内部的方法简称, 是由我们自己定义, 用来接收参数, 向后调用业务, 最终返回响应结果
  5. ViewResolver: SpringMVC 提供, 我们需要进行 IOC 配置使其加入 IOC 容器方可生效！视图解析器主要作用简化模板视图页面查找的, 但是在前后端分离项目中, 后端返回 JSON 数据, 不返回页面, 这个时候就不需要视图解析器。所有步骤8,9不是必须的

1.4 web 容器初始化原理讲解

前面的 SpringMvcInit 继承了 AbstractAnnotationConfigDispatcherServletInitializer

AbstractAnnotationConfigDispatcherServletInitializer 实现了 WebApplicationInitializer

WebApplicationInitializer 中有一个方法 onStartUp, 作用是 Web 项目启动时调用, 据此实现 Web 的IOC容器创建以及 DispatcherServlet 初始化

AbstractAnnotationConfigDispatcherServletInitializer 类中

==创建 service mapper 层容器== --> 通过 getRootConfigClasses() 获取配置类

@Override
@Nullable
protected WebApplicationContext createRootApplicationContext() {
    Class<?>[] configClasses = getRootConfigClasses(); 
    if (!ObjectUtils.isEmpty(configClasses)) {
        AnnotationConfigWebApplicationContext context = new AnnotationConfigWebApplicationContext(); // 根据配置类创建容器
        context.register(configClasses);
        return context;
    }
    else {
        return null;
    }
}

==创建 Web IOC 容器== --> 通过 getServletConfigClasses() 获取配置类

@Override
protected WebApplicationContext createServletApplicationContext() {
    AnnotationConfigWebApplicationContext context = new AnnotationConfigWebApplicationContext();
    Class<?>[] configClasses = getServletConfigClasses();
    if (!ObjectUtils.isEmpty(configClasses)) {
        context.register(configClasses);
    }
    return context;
}

所以说一共是有两个容器的

2. SpringMVC 接收数据

2.1 访问路径设置

@RequestMapping 注解的作用就是将请求的 URL 地址和处理请求的方式 (handler方法) 关联起来, 建立映射关系。将 handler 注册到 handlerMapping

SpringMVC 接收到指定的请求, 就会来找到在映射关系中对应的方法来处理这个请求

精准地址 [一个 | 多个]

@RequestMapping("/user/login"); // 一个地址
@RequestMapping(value = {"/user/login", "/login"}); // 多种访问地址都可以访问到

:tiger: 与 @WebServlet() 对比小细节

在 @WebServlet() 中要求地址必须是 / 开头, 即必须为 /login 才能访问到, 而对于 @RequestMapping, 则不要求, 即 /user/login == user/login == user/login/

支持模糊地址

  • * 指代任意一层字符串
    /user/* 则 /user/a , /user/aaa 都能访问到, /user/a/b 不行
  • ** 指代任意层任意字符串
    /user/** 则 /user, /user/a, /user/aa 都能访问到

类和方法上都能添加 @RequestMapping

@Target({ElementType.TYPE, ElementType.METHOD}) // 类和方法上都能用
public @interface RequestMapping {}

类上添加相当于在类中所有方法指定的地址上都加上前缀, 类地址 + 方法地址

@Controller
@RequestMapping("/user")
public class UserController {

    @RequestMapping // 加上 @RequestMapping 而不设置 value 值
    public String index(){ // "/user"
        return "index";
    }
    
    @RequestMapping("/login")
    public String login(){ // "/user/login"
        return null;
    }
    
    @RequestMapping("/register")
    public String register(){ // "/user/register"
        return null;
    }
}

请求方式指定

默认情况下(不指定 method), 只要地址正确, 任何请求方式都可以访问

指定请求方法 method = {RequestMethod.GET, RequestMethod.POST}

@RequestMapping(value = "/login", method = RequestMethod.GET)
public String login(){
    return null;
}
                                     // 数组形式, 指这两种访问方式都能访问到
@RequestMapping(value = "/register", method = {RequestMethod.GET, RequestMethod.POST})
public String register(){
    return null;
}

如果访问时请求方法不符合, 会出现 405 异常

注解进阶

@GetMapping("/login") 等价于 @RequestMapping(value = "/login", method = RequestMethod.GET)

不同点是 @GetMapping 只能作用于方法上, 类上不能用

同理还有 @PostMapping PutMapping DeleteMapping

2.2 接收参数

2.2.1 param 和 json 参数比较

在HTTP请求中, 我们可以选择不同的参数类型, 如 param 类型 (name=二狗&age=18) 和 JSON 类型 ({name:二狗,age:18})。以下是两种参数类型的区别

  1. 参数编码
    param 类型的参数会被编码为 ASCII 码。例如, name=john doe 会被编码为 name=john%20doe。而 JSON 类型的参数会被编码为 UTF-8。
  2. 参数顺序
    param 类型的参数没有顺序限制。但是, JSON 类型的参数是有序的。JSON采用键值对的形式进行传递, 其中键值对是有序排列的。
  3. 数据类型
    param 类型的参数仅支持字符串类型、数值类型和布尔类型等简单数据类型。而 JSON 类型的参数则支持更复杂的数据类型, 如数组、对象等。
  4. 嵌套性
    param 类型的参数不支持嵌套。但是, JSON 类型的参数支持嵌套, 可以传递更为复杂的数据结构。
  5. 可读性
    param 类型的参数格式比 JSON 类型的参数更加简单、易读。但是, JSON 格式在传递嵌套数据结构时更加清晰易懂。

总的来说, param 类型的参数适用于单一的数据传递, 而 JSON 类型的参数则更适用于更复杂的数据结构传递。根据具体的业务需求, 需要选择合适的参数类型。常见用法：在GET请求中参与 param 类型的参数, 而在 POST 请求中采用JSON类型的参数传递。

2.2.2 param 接收参数

直接接收

要求请求参数名 == 形参参数名, 参数可以不传递, 不会报错

// /param/data?name=二狗&age=18

@RequestMapping("/data")
@ResponseBody
public String data(String name, int age){
    return "name = " + name + ", age = " + age;
}

如果请求为 "/param/data?age=18", 则 name 会为 null。

但是我尝试这种写法, 并不能正常运行, 会报错, 具体原因未知, 还是需要在参数前加上注解才能运行

public String data(@RequestParam(value = "name", required = false) String name, @RequestParam(value = "age", required = false) Integer age){}

这里有一些细节需要注意

  1. 加上 @RequestParam 注解后, required 属性默认为 true, 即必须传递, 否则会报错
  2. 这里 age 如果为 int 类型, 并且不传递会报错, 但是如果为 Integer 类型, 不传递不会报错, 会为 null 值。这一点需要额外注意。

注解指定

可以指定任意的请求参数名, 可以要求参数必须传递或不必须传递, 还可以给定参数默认值

@Target(ElementType.PARAMETER)
@Retention(RetentionPolicy.RUNTIME)
@Documented
public @interface RequestParam {

  @AliasFor("name")
  String value() default "";

  @AliasFor("value")
  String name() default "";

  boolean required() default true;

  String defaultValue() default ValueConstants.DEFAULT_NONE;

}
  1. value 指定请求参数名 (文档中写如果形参名和请求参数名一致可以省略, 但是自己测试的是省略就会报错, 所以还是加上吧, 即使名字一致)
  2. required 默认是必须传递, 不穿会报 400 异常
  3. defaultValue 当非必须传递时来指定默认值
  4. 当设置了 value 值之后, 比如 @RequestParam(value = "account") String username, 则前端传递的参数中只能识别到 account 并赋值给参数列表中的 username, 而前端参数中的 username 并不会给参数列表中的 username。

使用注解进行指定

// 注解指定
@GetMapping("/data1")
public String data1(@RequestParam(value = "account") String username,
                    @RequestParam(value = "page", required = false, defaultValue = "1") int page){
    
    return "username = " + username + ", page = " + page;
}

请求1： "/param/data1?account=dog&page=2" 结果为 username = dog, page = 2
请求2： "/param/data1?account=dog" 结果为 username = dog, page = 1

注意到没, 这里使用 int 类型指定了默认值, 即使不传递也不会报错

特殊值 --> 一名多值

"/param/data2?hbs=sing&hbs=jump&hbs=rap", 用集合接收

@GetMapping("/data2")
public String data2(@RequestParam("hbs") List<String> hbs){
    return hbs.toString(); // [sing,jump,rap]
}

特殊值 --> 实体类

使用实体类接值, 此处定义了 User 类, 两个属性 name, age, 并添加了 getter 和 setter

需要保证前端参数列表与对象参数名称相同

"/param/data3?name="

@GetMapping("/data3")
public String data3(User user){
    return user.toString();
}
// User(name=dog, age=21)

2.2.3 接收路径参数

路径传递参数是一种 URL 路径中传递参数的方式。在 RESTful 的 Web 应用程序中, 经常使用路径传递参数来表示资源的唯一标识符或更复杂的表示方式。而 SpringMVC 框架提供了 @PathVariable 注解来处理路径传递参数。

@PathVariable 注解来处理路径传递参数, 允许将 URL 中的占位符映射到控制器方法中的参数。例如我们想将 /user/{id} 路径下的 {id} 映射到控制器方法的一个参数中, 则可以使用 @PathVariable 注解来实现。

设置动态路径

@RequestMapping("{参数1}/{参数2}")

接收路径参数, 通过 @PathVariable 接收！！必须加上这个注解且指定名字(相同也要指定)

实例程序中相同名字可以不指定 value, 但是我不指定会报错, 呜呜。

@Controller
@ResponseBody
@RequestMapping("path")
public class PathController {
  // path/dog/123
    @RequestMapping("{account}/{password}")
    public String login(@PathVariable("account") String username,
                        @PathVariable("password") String password){
        return username + " " + password; // dog 123
    }
}

2.2.4 json 数据接收

场景：接收如下 JSON 数据

创建 Person 实体类

@Data
public class Person {

    private String name;
    private int age;
    private String gender;

}

定义 Controller 进行接收, @RequestBody 指代接收 JSON 类型的参数

@Controller
@RequestMapping("/json")
@ResponseBody
public class JsonController {

    @PostMapping("data")
    public String data(@RequestBody Person person){
        System.out.println("person = " + person);
        return person.toString();
    }
}

发起请求 --> 报 415 异常

原因：Java原始的API只支持路径参数和param参数, 不支持json, json是前端的数据格式

解决：

  1. 导入 json 处理的依赖

    <dependency>
    <groupId>com.fasterxml.jackson.core</groupId>
    <artifactId>jackson-databind</artifactId>
    <version>2.15.3</version>
</dependency>

  2. handlerAdapter 配置 json 转换器 (因为是 handlerAdapter 负责数据的转换)
    在 MvcConfig 类上加入注解 @EnableWebMvc 即可

    @EnableWebMvc // 给 handlerAdapter 配置了 json 转换器
@Configuration
@ComponentScan("com.lh.json")
public class MvcConfig {

    @Bean
    public RequestMappingHandlerAdapter handlerAdapter(){
        return new RequestMappingHandlerAdapter();
    }

    @Bean
    public RequestMappingHandlerMapping handlerMapping(){
        return new RequestMappingHandlerMapping();

3.2.2 转发与重定向

转发使用 forward: 重定向使用 redirect:

@Controller
@RequestMapping("/jsp")
public class JspController {

    @GetMapping("/index")
    public String index(HttpServletRequest request){
        request.setAttribute("data", "hello jsp");
        return "index";
    }

    @GetMapping("/forward")
    public String forward(){
        System.out.println("JspController.forward");
        return "forward:/jsp/index";
    }

    @GetMapping("/forward/baidu")
    public String forwardBaidu(){
        System.out.println("JspController.forwardBaidu");
        return "forward:http://baidu.com"; // forward 不能访问项目外资源
    }

    @GetMapping("/redirect")
    public String redirect(){
        System.out.println("JspController.redirect");
        return "redirect:/jsp/index";
    }

    @GetMapping("/redirect/baidu")
    public String redirectBaidu(){
        System.out.println("JspController.redirectBaidu");
        return "redirect:http://baidu.com";
    }

}

下面将几个细节问题：

  1. redirect: 与 forward: 后面不能有空格, 直接连接内容即可, 比如 forward: /jsp/index 是不行的, : 后面有空格

  2. forward: 转发的是项目内的资源, 不能访问项目外的, 所以 "/jsp/forward/baidu" 是会报错的 redirect: 可以访问项目外的资源, 如百度。

  3. return "forward:/jsp/index"; 与 return "redirect:/jsp/index"; 与 return "/jsp/index"; 的区别

    return "/jsp/index"; 会经过视图解析器, 拼接为 /WEB-INF/views/jsp/index.jsp, 会去返回这个视图

    return "forward:/jsp/index"; 会去转发请求, 浏览器上的 url 地址不变, 但是实际会像 /jsp/index 发请求, 也就是我们定义的第一个 handler, 会返回 index.jsp 页面

    return "redirect:/jsp/index"; 会去重定向这个请求, 浏览器上的 url 地址会变为 /jsp/index, 然后发送请求, 也是会根据我们第一个 handler, 会返回 index.jsp 页面。

下面是五个 handler 的请求结果

第一个 "/jsp/index"

第二个 "/jsp/forward", url 地址不变, 但实际上处理是由 /jsp/index 来做的

第三个 "/jsp/forward/baidu", 不能请求项目外的资源, 所以报错

第四个 "/jsp/redirect", 重定向, url 地址改变为 /jsp/index

第五个 "/jsp/redirect/baidu", 重定向, 访问项目外资源, 直接到百度

一个路径细节问题 (可以不看)

项目通常会有项目上下文, 相当于设置一个项目名称。这里假设我们设置的 Application Context 是 /springmvc

  1. 不使用 springmvc 提供的, 而使用 request、response 进行重定向和转发

    转发是进行项目下的资源跳转, 路径: 项目下地址 /jsp/index, 会忽略 ApplicationContext, 它的设置不会影响转发的书写, 会默认帮我们加上

    重定向项目下的资源可以是项目外的资源, 所以需要我们手动将 ApplicationContext 加上, 如我们要进行重定向, 就必须写 /springmvc/jsp/index, /jsp/index 完成不了重定向

  2. 使用 springmvc 中的 forward: 与 redirect:

    转发和重定向都不需要加上 ApplicationContext 啦, 直接写转发或重定向的地址就可以了, 这样都是向项目内的资源进行转发或重定向, 如果想要重定向项目外的资源, 要用第五个 handler 的那种形式。

二者的区别在于重定向时需不需要加项目上下文的问题。

3.3.1 @ResponseBody

参数列表中的参数加上 @RequestBody 注解, 则 handlerAdapter 接收到 json 数据后会转为 java 对象, 当我们 return java对象; 时, handlerAdapter 会将 java对象转为 json。此时, 我们定义的 handler 上的 @ResponseBody 作用是不会查找视图解析器了, 而是放在响应体中直接返回。

@Controller
@RequestMapping("/json")
public class JsonController {

    @ResponseBody // 作用: 数据直接返回, 不找逻辑视图
    @GetMapping("/data")
    public User data(){
        User user = new User();
        user.setName("dog");
        user.setAge(3);
        return user; // user 由 handlerAdapter 转为 json
    }
}

返回结果:

{"name":"dog","age":3}

@ResponseBody 可以加在类上, 即类中所有的 handler 都将数据放入响应体返回, 不经过视图解析, 所以你的快速返回模板视图, 转发与重定向在此处都不能用了, 如果用 就给你当成纯字符串返回了。

@RestController == @Controller + @ResponseBody

所以如果类中所有方法都返回数据, 可以为

现在有如下需求, 访问静态资源目录

解释：WEB-INF 下的资源是不能通过浏览器 url 访问到的, 属于敏感资源, 而对于非 WEB-INF 下的目录资源是可以通过浏览器 url 访问到的。
现在我们尝试通过浏览器访问 http://localhost:8080/images/picture.jpg , 结果报 404 错误。

原因是我们所有的请求都会经过 DispatcherServlet 进行分发, 它在 handlerMapping 中并没有找到 /images/picture.jpg 对应的 handler, 所以会报错。现在我们修改以下代码, 添加这种 handler 进行验证

虽然请求有点怪异, 但是确实可以访问到。

其实要访问静态资源, 需要我们手动开启以下静态资源访问, 在 MvcConfig 配置类中

@Configuration
@ComponentScan("com.lh.json")
@EnableWebMvc
public class MvcConfig implements WebMvcConfigurer {

    // 视图解析器, 指定前后缀
    @Override
    public void configureViewResolvers(ViewResolverRegistry registry) {
        // registry 快速添加前后缀
        registry.jsp("/WEB-INF/views/", ".jsp");
    }

    // 开启静态资源访问, 如果在 handlerMapping 中找不到, 则去静态资源中查找
    @Override
    public void configureDefaultServletHandling(DefaultServletHandlerConfigurer configurer) {
        configurer.enable(); // 开启
    }
}

然后我们为了测验是不是先去找 handler 找不到再去静态资源中找, 那个怪异的 handler 先留着, 我们访问 "/images/picture.jpg" 时确实还是那个 handler 返回的结果

然后我们把这个 handler 删除, 再访问一下 "/images/picture.jpg" 是可以访问到的！！

4. RestFul 风格设计和实战

4.1 介绍

RestFul -> Http 协议的标准使用方案和风格

  • 教你如何设计路径
  • 教你如何设计参数传递
  • 教你如何选择请求方式

在进行 url 请求时, 我们需要给出如下三个内容

  1. url 地址
  2. 请求方式 get put delete post
  3. 传递参数的方式 param json path (路径传参)

问题引入

当我们进行用户增删改查的时候

原来需要写 /user/add, /user/delete 等表示我这个 handler 进行用户添加和删除, 然后指定方式 get post 其他几种都没用过, 传递参数的形式也要规定好使用那种。

4.1.1 RestFul 风格特点

  1. 每一个 URI 代表 1 种资源 (URI 是名词) --> URL 是 URI 的一种, 具体区别暂时不管
    即原来书写的 URL 是 /user/add, /user/delete 动词也加入了 URL 中, 这种是不合理的, 因为 http 设计者已经给出了四种(实际更多)请求方式进行增删改查。

    所以原来的老方法根据 id 添加用户 /user/add?id=1 根据 id 添加用户 /user/delete?id=1 根据 id 删除用户。(请求方式 get 和 post 都是随便选择)

    规整后, 应该是 URL 中不含动词, 添加用户 /user/1 使用 post 方式请求, 删除用户 /user/1 使用 delete 方式请求

  2. 客户端使用 GET 获取资源、POST 来新建资源(也可以更新)、PUT 来更新资源、DELETE 来删除资源

  3. 资源的表现形式是 XML 或者 ==JSON==。

URI 统一资源标识符, URL 统一资源定位符。

URL 路径风格要求如下

操作 传统风格 REST风格
保存 /CRUD/saveEmp URL请求地址: /CRUD/emp
请求方式: POST
删除 /CRUD/removeEmp?empId=2 URL请求地址: /CRUD/emp/2
请求方式: DELETE
更新 /CRUD/udpateEmp URL请求地址: /CRUD/emp
请求方式: PUT
查询 /CRUD/editEmp?empId=2 URL请求地址: /CRUD/emp/2
请求方式: GET

具体动作选择具体的HTTP协议请求方式, 路径设计从原来携带动词改为名词

优点：简洁、优雅, 安全(通过URL地址不知道是什么具体动作了, param 参数也不知道是什么含义了)

4.2 RestFul 风格实战

4.2.1 需求分析

  • 数据结构: User {id, name, age}

  • 功能分析

    用户数据分页展示功能 (条件: page 页数 默认1, size 每页数量 默认10)

    保存用户功能
    根据用户 id 查询用户详情功能
    根据用户 id 更新用户数据功能
    根据用户 id 删除用户数据功能
    多条件模糊查询用户功能 (条件: keyword 模糊关键字, page 页数 默认1, size 每页数量 默认10)

4.2.2 RestFul 风格接口设计

功能 接口和请求方式 请求参数 返回值
分页查询 GET /user param page=1&size=10 {响应数据}
用户添加 POST /user {user 数据} {响应数据}
用户详情 GET /user/1 路径参数 {响应数据}
用户更新 PUT /user {user 更新数据} {响应数据}
用户删除 DELETE /user/1 路径参数 {响应数据}
模糊查询 GET /user/search page=1&size=10&keyword=关键字 {响应数据}

误区: restful 风格下, 不是所有的请求参数都是路径传参! 可以使用其他方式传递

:question: 具体使用哪种方式呢

首先我们要知道 GET、DELETE 是没有请求体的, 所以只能通过 param 参数和路径参数 进行传参。而 PUT 和 POST 可以使用这三种方式

  • PUT, POST 一般直接使用请求体传参即可, 即 JSON 数据
  • GET, DELETE 如果使用路径参数一般是对一个数据进行操作, 如根据 id, 如果是范围性的操作 (操控多个数据), 使用 param 形式, 因为路径参数的组合和排列可能会很多, 不如使用请求参数(key=value形式)更加灵活和简洁。

注意

  • 对于敏感数据, 最好使用 POST+请求体的方式
  • 请求参数应该限制在10个以内, 便于接口的维护和使用
  • 以上仅是标准, 一种风格, 尽量遵守, 如果有更好的方式也可以不使用这种哦

5. SpringMVC 其他扩展

5.1 全局异常处理机制

5.1.1 异常处理两种方式

对于异常的处理, 通常有两种方式

  • 编程式异常处理：即哪里可能有错, 哪里用 try-catch-finally, 不同位置的同一种错误要重复处理, 且异常处理代码混杂在业务代码中, 导致代码可读性差
  • 声明式异常处理
    将异常处理的逻辑从具体的业务逻辑中分离出来, 通过配置等方式进行统一的管理和处理。在声明式异常处理中, 开发人员只需要为方法或类标注相应的注解 (如 @Throws 或 @ExceptionHandler), 就可以处理特定类型的异常。

一个项目通常包含多个模块, 各个模块需要分工完成, 当对于同一种异常, 不同人的处理方案不同时, 整个项目就会变得混乱, 所以声明式异常处理显然更加易于维护和扩展。

5.1.2 基于注解异常声明和异常处理

首先在 controller 中搞两个异常

@RestController
@RequestMapping("/error")
public class UserController {

    @GetMapping("/data")
    public String data(){
        String name = null;
        name.toString(); // NullPointerException
        return "ok";
    }

    @GetMapping("/data2")
    public String data2(){
        int i = 1 / 0; // ArithmeticException:
        return "ok";
    }
}

正常情况下我们访问后会出现错误

如果我们直接在这些可能出现的错误处使用 try-catch-finally, 那就是编程式异常处理, 而我们需要使用声明式异常处理。

第一步：==声明异常处理控制器类==

异常处理控制类, 统一定义异常处理 handler 方法

// @ControllerAdvice
@RestControllerAdvice
public class GlobalExceptionHandler {  
    
}

需要添加注解进行声明, 两个只能选择一个

  • @ControllerAdvice 返回的内容将通过视图解析器, 返回的结果是页面
  • @RestControllerAdvice 返回的是 JSON 字符串

第二部：==声明异常处理 handler 方法==

@RestControllerAdvice
public class GlobalExceptionHandler {

    // 异常类型
    @ExceptionHandler(ArithmeticException.class)
    public Object ArithmeticExceptionHandler(ArithmeticException e){
        String message = e.getMessage();
        System.out.println("message = " + message);

        return message;
    }

    @ExceptionHandler(Exception.class)
    public Object ExceptionHandler(Exception e){
        String message = e.getMessage();
        System.out.println("message = " + message);

        return message;
    }

}

注意命名方法:

@ExceptionHandler(Exception.class)
public Object ExceptionHandler(Exception e) // 写异常名称的三个地方一致就可以

当异常没有找到时, 会去找父类异常, 如此处没有 NullPointerException 异常的处理, 就会向上找到 ExceptionHandler 处理异常。

第三步：==将异常处理的类放入容器==

@Configuration
@ComponentScan({"com.lh.controller", "com.lh.error"}) // 扫描包即可
@EnableWebMvc
public class MvcConfig implements WebMvcConfigurer {

}

运行查看结果

  1. 当使用 @ControllerAdvice 注解时, 我们可以用来返回页面 (比如一个好看的 404 页面)
  2. 当使用 @RestControllerAdvice 注解时, 我们可以向前端返回错误信息(出错原因, 出错的code等等)

这种异常处理底层实际是使用 AOP 实现的。

5.2 拦截器

5.2.1 拦截器概念

拦截器 VS 过滤器

拦截器是 springmvc 提供的, 而过滤器是 javaweb 提供的 (javaweb项目都能用)

:tiger: 相似点

  • 拦截：必须先把请求拦住, 才能执行后续操作
  • 过滤：拦截器或过滤器存在的意义就是对请求进行统一处理
  • 放行：对请求执行了必要操作后, 放请求过去, 让它访问原本想要访问的资源

:tiger: 不同点

  • 工作平台不同
    过滤器工作在 Servlet 容器中, 拦截器工作在 SpringMVC 的基础上
  • 拦截的范围
    过滤器能够拦截到的最大范围是整个Web应用, 拦截器能够拦截到的最大范围是整个SpringMVC负责的请求
  • IOC容器支持
    过滤器：想得到IOC容器需要调用专门的工具方法, 是间接的
    拦截器：它自己就在IOC容器中, 所以可以直接从IOC容器中装配组件, 也就是可以直接得到IOC容器的支持

所以说如果用SpringMVC的拦截器能够实现的功能, 就不使用过滤器

5.2.2 拦截器的使用

步骤1：==创建拦截器类==

三个方法都有默认实现, 所以只需要根据需要重写方法即可。

package com.lh.interceptor

public class MyInterInterceptor implements HandlerInterceptor {

    // 执行 handler 之前调用的拦截方法
    // 编码格式设置, 登录保护, 权限处理

    /**
     * @param request 请求对象
     * @param response 响应对象
     * @param handler handler就是我们要调用的方法对象
     * @return true 放行 false 拦截
     * @throws Exception
     */
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
      return true;
    }

    /**
     * 当 handler 执行完毕后触发的方法
     * 通常用来对结果处理, 敏感词汇检查!
     * @param request 请求对象
     * @param response 响应对象
     * @param handler handler方法
     * @param modelAndView 返回的视图和共享域数据对象
     * @throws Exception
     */
    @Override
    public void postHandle(HttpServletRequest request, HttpServletResponse response, Object handler, ModelAndView modelAndView) throws Exception {
        
    }

    /**
     * 整体处理完毕
     * @param request 请求对象
     * @param response 响应对象
     * @param handler handler就是我们要调用的方法对象
     * @param ex handler报错时的异常对象
     * @throws Exception
     */
    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) throws Exception {
        
    }
}

当拦截器中的 preHandle 返回 false 时即拦截, 那么后续方法的 postHandle 和 afterCompletion 都不会执行啦

步骤2：==在MvcConfig中加入拦截器==

重写 addInterceptors 方法

@Configuration
@ComponentScan({"com.lh.controller", "com.lh.error"})
@EnableWebMvc
public class MvcConfig implements WebMvcConfigurer {

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        // 默认拦截全部请求
        registry.addInterceptor(new MyInterInterceptor()); // 加入拦截器
    }
}

拦截器细节配置

指定地址拦截：通过链式调用

@Override
public void addInterceptors(InterceptorRegistry registry) {
  
    registry.addInterceptor(new MyInterInterceptor())
            .addPathPatterns("/user/data"); // 只拦截 /user/data 请求
}
// 支持 [*|**] 方式
registry.addInterceptor(new MyInterInterceptor())
    .addPathPatterns("/user/**")
    .excludePathPatterns("/user/data"); // 拦截 /user/下的所有请求, 除了 /user/data

5.2.3 拦截器优先级

先声明的优先级高, 优先级高的在外层

registry.addInterceptor(new MyInterceptor());
registry.addInterceptor(new MyInterceptor1());

顺序如下

MyInterceptor.preHandle
MyInterceptor1.preHandle
OrderController.data
MyInterceptor1.postHandle
MyInterceptor.postHandle
MyInterceptor1.afterCompletion
MyInterceptor.afterCompletion
