进程与线程

进程

进程：程序是静止的，进程实体的运行过程就是进程，是系统进行资源分配的基本单位

  • 程序由指令和数据组成，但这些指令要运行，数据要读写，就必须将指令加载至 CPU，数据加载至内存。在指令运行过程中还需要用到磁盘、网络等设备。进程就是用来加载指令、管理内存、管理 IO 的。
  • 当一个程序被运行，从磁盘加载这个程序的代码至内存，这时就开启了一个进程。
  • 进程就可以视为程序的一个实例。大部分程序可以同时运行多个实例进程（例如记事本、画图、浏览器等，比如开启多个记事本，浏览器），也有的程序只能启动一个实例进程（例如网易云音乐、360 安全卫士等）。

进程的特征：并发性、异步性、动态性、独立性、结构性

线程

线程：线程是属于进程的，是一个基本的 CPU 执行单元，是程序执行流的最小单元。线程是进程中的一个实体，是系统独立调度的基本单位，线程本身不拥有系统资源，只拥有一点在运行中必不可少的资源，与同属一个进程的其他线程共享进程所拥有的全部资源

  • 一个进程之内可以分为一到多个线程。

  • 一个线程就是一个指令流，将指令流中的一条条指令以一定的顺序交给 CPU 执行。

  • Java 中，线程作为最小调度单位，进程作为资源分配的最小单位。在 windows 中进程是不活动的，只是作为线程的容器。

关系：一个进程可以包含多个线程，这就是多线程，比如看视频是进程，图画、声音、广告等就是多个线程

线程的作用：使多道程序更好的并发执行，提高资源利用率和系统吞吐量，增强操作系统的并发性能

并发并行

  • 并行（parallel）：是同一时间动手做（doing）多件事情的能力。
    多核CPU下，每个核都可以调度运行线程，这个时候线程可以是并行的。要运行的线程数一般会比CPU核数多，所以也会出现并发（这个时候也会有竞争，比如两个线程都要用同一个资源，那么其中一个就得等待另一个用完）

  • 并发（concurrent）：是同一时间应对（dealing with）多件事情的能力，这些事不一定在同一时刻运行，比如边炒菜边做饭，中间来回交替，是在并发，但不是并行，因为一个人只能在炒菜或者做饭。

    单核 cpu 下，线程实际还是==串行执行==的。操作系统中有一个组件叫做任务调度器，将 cpu 的时间片（windows 下时间片最小约为 15 毫秒）分给不同的线程使用，只是由于 cpu 在线程间（时间片很短）的切换非常快，人类感觉是同时运行的。总结为一句话就是：微观串行，宏观并行，

一般会将这种 线程轮流使用 CPU 的做法称为并发 concurrent

其实单核CPU下，假如有四个线程，如果并发会比串行执行更耗时，因为有上下文切换耗时。

  1. 单核 cpu 下，多线程不能实际提高程序运行效率，只是为了能够在不同的任务之间切换，不同线程轮流使用 cpu，不至于一个线程总占用 cpu，别的线程没法干活
  2. 多核 cpu 可以并行跑多个线程，但能否提高程序运行效率还是要分情况的
    • 有些任务，经过精心设计，将任务拆分，并行执行，当然可以提高程序的运行效率。但不是所有计算任务都能拆分（参考后文的【阿姆达尔定律】）
    • 也不是所有任务都需要拆分，任务的目的如果不同，谈拆分和效率没啥意义
  3. IO 操作不占用 cpu，只是我们一般拷贝文件使用的是【阻塞 IO】，这时相当于线程虽然不用 cpu，但需要一直等待 IO 结束，没能充分利用线程。所以才有后面的【非阻塞 IO】和【异步 IO】优化

对比

线程进程对比：

  • 进程基本上相互独立的，而线程存在于进程内，是进程的一个子集

  • 进程拥有共享的资源，如内存空间等，供其内部的线程共享

  • 进程间通信较为复，同一台计算机的进程通信称为 IPC（Inter-process communication）。不同计算机之间的进程通信，需要通过网络，并遵守共同的协议，例如 HTTP

  • 线程通信相对简单，因为线程之间共享进程内的内存，一个例子是多个线程可以访问同一个共享变量

    Java 中的通信机制：volatile、等待/通知机制、join 方式、InheritableThreadLocal、MappedByteBuffer

  • 线程更轻量，线程上下文切换成本一般上要比进程上下文切换低

应用

同步异步：

  • 需要等待结果返回，才能继续运行就是同步
  • 不需要等待结果返回，就能继续运行就是异步

多线程可以让方法执行变为异步的。比如说读取磁盘文件时，假设读取操作花费了5秒钟，如果没有线程调度机制，这5秒调用者什么也做不了，其代码都得暂停。所以实际开发钟，如果某些操作，比如视频文件需要转换格式等，这些操作比较费事，就需要开一个新线程，避免阻塞主线程。

多线程可以利用多核CPU，如果只是顺序执行，那不就只能用一个CPU。

Java 线程

创建线程

Java 运行的时候默认就有一个主线程了，就是 main，如果想在这个之外再创建线程，得用下面这些方法

Thread

  • start() 方法底层其实是给 CPU 注册当前线程，并且触发 run() 方法执行
  • 线程的启动必须调用 start() 方法，因为如果线程直接调用 run() 方法，相当于变成了调用对象的方法执行，跟线程没关系了
  • 建议线程先创建子线程，主线程的任务放在之后，因为主线程是顺序执行的，可以先执行子线程去跑，然后再跑主线程（main）

Thread 类本身也是实现了 Runnable 接口，Thread 类中持有 Runnable 的属性，执行线程 run 方法底层是调用 Runnable#run：

// 源码
public class Thread implements Runnable {
    private Runnable target;
    
    // 重写的 Runnable 里面的 run
    @Override
    public void run() {
        if (target != null) {
          	// 底层调用的是 Runnable 的 run 方法
            target.run();
        }
    }
}

第一种 Thread 方法其实就是重写了 Thread 里面的 run() 方法。

Runnable 方式的优缺点：

  • 缺点：代码复杂一点。
  • 优点：
    1. Thread 是把线程和任务合并在了一起，Runnable 是把线程和任务分开了
    2. 用 Runnable 更容易与线程池等高级API配合
    3. 用 Runnable 让任务类脱离了 Thread 继承体系，更灵活

Callable

// The Callable interface is similar to Runnable, in that both are designed for classes whose instances are potentially executed by another thread. A Runnable, however, does not return a result and cannot throw a checked exception.
@FunctionalInterface
public interface Callable<V> {

    V call() throws Exception;
}

和 Runnable 类似，但是它有返回值并且能抛出异常

public class ThreadDemo {
    public static void main(String[] args) {
        Callable call = new MyCallable(); // 定义任务执行+返回值
        FutureTask<String> task = new FutureTask<>(call); // 任务
        Thread t = new Thread(task); // 创建线程
        t.start();
        try {
            String s = task.get(); // 获取call方法返回的结果（正常/异常结果）在这里阻塞等待结果
            System.out.println(s);
        }  catch (Exception e) {
            e.printStackTrace();
        }
    }

public class MyCallable implements Callable<String> {
    @Override//重写线程任务类方法
    public String call() throws Exception {
        return Thread.currentThread().getName() + "->" + "Hello World";
    }
}

这里的 FutureTask 可以通过 task.get() 方法来决定什么时候获取返回的结果，只能读取一次哦。FutureTask 可以把线程任务对象包装成一个==未来任务对象==，因为它继承了 Runnable 对象，所以也可以作为 Thread 的构造参数。

public FutureTask(Callable<V> callable)：未来任务对象，在线程执行完后得到线程的执行结果

  • FutureTask 就是 Runnable 对象，因为 Thread 类只能执行 Runnable 实例的任务对象，所以把 Callable 包装成未来任务对象
  • 线程池部分详解了 FutureTask 的源码

public V get()：同步等待 task 执行完毕的结果，如果在线程中获取另一个线程执行结果，==会阻塞等待==，用于线程同步

  • get() 线程会阻塞等待任务执行完成
  • run() 执行完后会把结果设置到 FutureTask 的一个成员变量，get() 线程可以获取到该变量的值
    具体可以看 FutureTask 的 run 方法，它把 Callable.call() 方法执行完的结果存储起来了，存到 outcome 变量里面。

优缺点：

  • 优点：同 Runnable，并且能得到线程执行的结果
  • 缺点：编码复杂

为什么不像 Runnable 那样，直接作为参数传递到 Thread，而是在这里又借助了 FutureTask？

  1. Thread 和 Runnable 在 JDK1.0就有了，Thread 构造方法设计死了一个标准，就是只接收 Runnable 接口，但是 Callable 在 JDK1.5才有，为了向下兼容，，，FutureTask 类被实现来解决这个问题，继承了 Runnable。
  2. Callable 和 Runnable 不同的是它有返回值，所以线程执行完之后，还需要一个地方存放返回值（最核心的原因），所以有 FutureTask，它的 run 方法调用 Callable 的 call 方法并且把返回值存储起来。

线程运行原理

栈与栈帧

Java Virtual Machine Stacks（Java 虚拟机栈）

我们都知道JVM由堆、栈、方法区所组成，其中栈内存是给谁用的呢？其实就是线程，每个线程启动后，虚拟机就会为其分配一块栈内存。

  • 每个栈由多个栈帧（Frame）组成，对应着每次方法调用时所占用的内存
  • 每个线程只能有一个活动栈帧，对应着当前正在执行的那个方法
public class TestFrames {

    public static void main(String[] args) {
        method1(10);
    }

    private static void method1(int x) {
        int y = x + 1;
        Object m = method2();
        System.out.println(m);
    }

    private static Object method2() {
        Object o = new Object();
        return o;
    }
}

main 函数执行，就是一个主线程，会被分配到一块栈内存，刚开始就是 main，他自己也是一个栈帧，当往下执行到 method1 的时候，method1 作为一个新栈帧加入栈内存，当进入 method2 又是一个新的栈帧，跑完删除，这就不多说了。

栈帧图解

比如如上程序，当程序运行时，把这个 TestFrames 类字节码加载到虚拟机中，存放在方法区。Java虚拟机会启动一个 main 的主线程，并且分配一块线程栈。然后main栈帧就加载到这个里面，其中args是一个字符串数组，对象实体放在堆中，args指向那里。当执行到 method1 时，创建 method1 的栈帧，并且返回地址也就是调用 method1 方法的下一条指令地址。这里程序计数器里面放的是当前要执行的二进制指令，这里为了简化方便看，直接放的代码。CPU查看程序计数器的指令进行执行，比如这里程序计数器放的是 new Object()，那么就会去堆中申请内存，放入对象，并且赋值给局部变量 n。

线程上下文切换（Thread Context Switch）

一些原因导致 CPU 不再执行当前线程，转而执行另一个线程

  • 线程的 CPU 时间片用完
  • 垃圾回收（所有工作线程都要暂停，进行垃圾回收）
  • 有更高优先级的线程需要运行
  • 线程自己调用了 sleep、yield、wait、join、park、synchronized、lock 等方法

前三种都是被动的，第四种是主动暂停自己的线程，让其他线程来运行的。

程序计数器（Program Counter Register）：记住下一条 JVM 指令的执行地址，==是线程私有的==，因为每个线程都有自己要执行的程序，所以每个线程都会有一个。

当 Context Switch 发生时，需要由操作系统保存当前线程的状态（PCB 中），并恢复另一个线程的状态

  • 状态包括程序计数器、虚拟机栈中每个栈帧的信息，如局部变量、操作数栈、返回地址等

  • Context Switch 频繁发生会影响性能，所以线程数不是越多越好，当线程数大于核数的时候，必然会进行线程切换，当大于很多的时候，切换频率会很高。

比如这个图，在上下文切换时，CPU 会将当前线程的程序计数器（PC）和寄存器快照保存到线程控制块（TCB）中；而该线程在内存（JVM Stack）里的各级栈帧信息保持不动，待下次切换回该线程时，再通过恢复 PC 指针和寄存器值来延续之前的执行现场。

start 与 run

start 是启动线程，run 是线程启动以后要执行的代码。

:question: 执行 start 方法和执行 run 方法有什么区别呢？

public static void main(String[] args) {

    Thread t1 = new Thread(() -> {
        log.debug("running...");
    }, "t1");

    t1.run(); // run
    log.debug("running...");
}

public static void main(String[] args) {

    Thread t1 = new Thread(() -> {
        log.debug("running...");
    }, "t1");

    t1.start(); // start
    log.debug("running...");
}

对比 run 和 start 方法运行结果，其实第一个用 run 并没有开启线程，你看输出的两个 running... 都是在 main 线程里面，因为这种调用方式相当于调用普通对象的方法，所以还是在主线程里面。

然后我们看一下 start 前后线程的状态

public static void main(String[] args) {

        log.debug("running...");
    }, "t1");

    System.out.println(t1.getState());
    t1.start();
    System.out.println(t1.getState());
}

start 之前线程是新建的状态，还没有被CPU运行，线程启动以后是 RUNNABLE，代表可以被CPU调度执行了。

不能重复调用 start 方法，否则会报错

sleep 与 yield

sleep：

  1. 调用 sleep 会让当前线程从 Running 进入 Timed Waiting 状态（阻塞）

  2. 其它线程可以使用 interrupt 方法打断正在睡眠的线程，这时 sleep 方法会抛出 InterruptedException

  3. 睡眠结束后的线程未必会立刻得到执行，需要抢占 CPU

  4. 建议用 TimeUnit 的 sleep 代替 Thread 的 sleep 来获得更好的可读性

yield：

  1. 调用 yield 会让当前线程从 Running 进入 Runnable 就绪状态，然后调度执行其他线程
  2. 具体的实现依赖于操作系统的任务调度器（比如只有当前一个线程，yield 后没有别的线程了，就还是让当前线程运行）

不同：任务调度器会给就绪状态的线程分配时间片，但是阻塞状态的不会；并且 yield 没有参数，基本上没有等待时间，但是 sleep 会有个时间参数，会有一个等待时间。

sleep

我们看看状态变化，线程阻塞时候的状态标志，也就是刚刚的第一条，调用 sleep 会让当前线程从 Running 进入 Timed Waiting 状态。

public static void main(String[] args) throws InterruptedException {
    Thread t1 =  new Thread("t1"){
        public void run() {
            try {
                Thread.sleep(2000);
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            }
        }
    };

    t1.start();
    log.debug("t1 state: {}", t1.getState()); // RUNNABLE

    Thread.sleep(500);
    log.debug("t1 state: {}", t1.getState()); // TIMED_WAITING
}

Thread.sleep() 在哪个线程被调用就是让哪个线程休眠。

然后我们演示第二条：其它线程可以使用 interrupt 方法打断正在睡眠的线程，这时 sleep 方法会抛出 InterruptedException

public static void main(String[] args) throws InterruptedException {

    Thread t1 = new Thread(() -> {
        log.debug("enter sleep...");
        try {
            Thread.sleep(2000);
        } catch (InterruptedException e) {
            log.debug("wake up...");
            e.printStackTrace(); // 红色字是这里打印输出的信息
        }
    }, "t1");
    t1.start();

    Thread.sleep(1000);
    log.debug("interrupt...");
    t1.interrupt(); // 打断 t1 的睡眠
}

interrupt

打断 sleep、wait、join 的线程

这三种方法会阻塞线程，在打断这三种方法会响应中断并抛出异常，同时JVM还是强制将中断标记位重置为 false。

public static void test1() throws InterruptedException {
    Thread t1 = new Thread(() -> {
        log.debug("sleep...");
        try {
            Thread.sleep(5000); // 睡眠 5 秒
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }, "t1");

    t1.start();
    Thread.sleep(1000); // 主线程睡眠 1 秒，等待线程进入睡眠
    log.debug("interrupt");
    t1.interrupt();
    log.debug("打断标记:{}", t1.isInterrupted());
}

子线程进入睡眠后，主线程对子线程进行打断。但是这里打断标记并没有像预设的那样打断标记为 false 啊

原因是主线程和子线程在竞速，当代码运行到 t1.interrupt() 之后

主线程：调用底层内核方法，把 t1 线程的 TCB 标记位改为 true，下一秒立即向下执行

t1 线程：从睡眠中唤醒，恢复寄存器，抛出异常，抛出异常前一刻，JVM 执行指令将标记位重置为 false

所以主线程速度更快一点。

模式之两阶段终止

Two Phase Termination

在一个线程 T1 中如何优雅终止线程 T2？这里的优雅是指给 T2 一个料理后事的机会

错误思路

  1. 使用线程对象的 stop() 方法停止线程
    stop 方法会真正杀死线程，如果这时线程锁住了共享资源，那么当它被杀死后就再也没有机会释放锁，其他线程将永远无法获取锁
  2. 使用 System.exit(int) 方法停止线程
    目的仅是停止一个线程，但这种做法会让整个程序都停止

两阶段终止模式

比如这个监控记录，正常情况下，判断有没有被打断，没有，就睡眠两秒，然后执行监控记录，意思就是两秒监控一次（因为监控没必要每秒都盯着）；如果被打断了，就料理后事，然后结束循环。如果说在正常执行监控记录的时候被打断了，那就是打断标记被设置为 true 了，然后下一轮循环的时候就可以被判断为打断了然后料理后事结束循环。如果说睡眠的时候被打断了，这个时候会抛出异常，会清空打断标记，这个时候我们在异常处理里面要重新设置打断标记。

代码如下

@Slf4j(topic = "c.Test10")
public class Test10 {
    public static void main(String[] args) throws InterruptedException {
        TwoPhaseTermination tpt = new TwoPhaseTermination();
        tpt.start();

        Thread.sleep(3500); // 睡眠3.5秒后停止监控
        tpt.stop();
    }
}

@Slf4j(topic = "c.TwoPhaseTermination")
class TwoPhaseTermination{
    private Thread monitor;

    // 启动监控线程
    public void start() {
        monitor = new Thread(() -> {
            while (true) {
                Thread current = Thread.currentThread();
                if(current.isInterrupted()){
                    log.debug("料理后事");
                    break;
                }
                try {
                    Thread.sleep(1000);
                    log.debug("执行监控记录");
                } catch (InterruptedException e) {
                    e.printStackTrace();
                    // 重新设置打断标记
                    current.interrupt();
                }
            }
        });
        monitor.start();
    }

    // 停止监控线程
    public void stop() {
        monitor.interrupt();
    }
}

另外注意 isInterrupted() 判断是否打断不会清除打断标记。interrupted() 判断当前线程是否被打断，会清除打断标记。

        log.debug("打断状态: {}", Thread.interrupted()); // 重置打断标记

        LockSupport.park(); // 再来一次 park
        log.debug("unpark...");
    }, "t1");

    t1.start();
    sleep(1000);
    t1.interrupt();

线程状态

五种状态

这是从操作系统层面来描述的

  • 【初始状态】 仅是在语言层面创建了线程对象，还未与操作系统线程关联

  • 【可运行状态】（就绪状态）指该线程已经被创建（与操作系统线程关联），可以由 CPU 调度执行

  • 【运行状态】 指获取了 CPU 时间片运行中的状态

    • 当 CPU 时间片用完，会从 【运行状态】 转换至 【可运行状态】，会导致线程的上下文切换

  • 【阻塞状态】

    • 如果调用了阻塞 API，如 BIO 读写文件，这时该线程实际不会用到 CPU，会导致线程上下文切换，进入 【阻塞状态】

    • 等 BIO 操作完毕，会由操作系统唤醒阻塞的线程，转换至 【可运行状态】

    • 与 【可运行状态】 的区别是，对 【阻塞状态】 的线程来说只要它们一直不唤醒，调度器就一直不会考虑调度它们

  • 【终止状态】 表示线程已经执行完毕，生命周期已经结束，不会再转换为其它状态

六种状态

这个是从 Java API 层面来说的，根据 Thread.State 枚举，分为 6 种状态

  • NEW 线程刚被创建，但是还没有调用 start() 方法

  • RUNNABLE 也就是调用了 start() 方法之后。注意，Java API 层面的 RUNNABLE 状态涵盖了 操作系统 层面的【可运行状态】、【运行状态】和【阻塞状态】（由于 BIO 导致的线程阻塞，在 Java 里无法区分，仍然认为是可运行）

  • BLOCKED, WAITING, TIMED_WAITING 都是 Java API 层面对【阻塞状态】的细分，后面会在状态转换一节详述

  • TERMINATED 当线程代码运行结束

线程状态 导致状态发生条件
NEW（新建） 线程刚被创建，但是并未启动，还没调用 start 方法，只有线程对象，没有线程特征
Runnable（可运行） 线程可以在 Java 虚拟机中运行的状态，可能正在运行自己代码，也可能没有，这取决于操作系统处理器，调用了 t.start() 方法：就绪（经典叫法）
Blocked（阻塞） 当一个线程试图获取一个对象锁，而该对象锁被其他的线程持有，则该线程进入 Blocked 状态；当该线程持有锁时，该线程将变成 Runnable 状态
Waiting（无限等待） 一个线程在等待另一个线程执行一个（唤醒）动作时，该线程进入 Waiting 状态，进入这个状态后不能自动唤醒，必须等待另一个线程调用 notify 或者 notifyAll 方法才能唤醒
Timed Waiting （限期等待） 有几个方法有超时参数，调用将进入 Timed Waiting 状态，这一状态将一直保持到超时期满或者接收到唤醒通知。带有超时参数的常用方法有 Thread.sleep 、Object.wait
Teminated（结束） run 方法正常退出而死亡，或者因为没有捕获的异常终止了 run 方法而死亡

临界区

  • 一个程序运行多个线程本身是没有问题的
  • 问题出在多个线程访问共享资源
    • 多个线程读共享资源其实也没有问题
    • 在多个线程对共享资源读写操作时发生指令交错，就会出现问题
  • 一段代码块内如果存在对共享资源的多线程读写操作，称这段代码块为临界区

例如，下面代码中的临界区

static int counter = 0;
static void increment() 
// 临界区
{ 
 counter++;
}
static void decrement() 
// 临界区
{ 
 counter--;
}

count++ 和 count-- 都包含读+写的操作，所以这一个操作包含了读写，counter 是共享资源，所以单独的 { count++; } { count--; } 是临界区。

竞态条件

多个线程在临界区内执行，由于代码的执行序列不同而导致结果无法预测，称之为发生了竞态条件

synchronized 解决方案

* 应用之互斥

为了避免临界区的竞态条件发生，有多种手段可以达到目的。

  • 阻塞式的解决方案：synchronized，Lock
  • 非阻塞式的解决方案：原子变量

本章使用阻塞式的解决方案：synchronized，来解决上述问题，即俗称的【对象锁】，它采用互斥的方式让同一
时刻至多只有一个线程能持有【对象锁】，其它线程再想获取这个【对象锁】时就会阻塞住。这样就能保证拥有锁
的线程可以安全的执行临界区内的代码，不用担心线程上下文切换

虽然 java 中互斥和同步都可以采用 synchronized 关键字来完成，但它们还是有区别的：

  • 互斥是保证临界区的竞态条件发生，同一时刻只能有一个线程执行临界区代码
  • 同步是由于线程执行的先后、顺序不同、需要一个线程等待其它线程运行到某个点

语法

{
    临界区
}

synchronized 理解

可以做这样的类比：

  • synchronized(对象) 中的对象，可以想象为一个房间（room），有唯一入口（门），房间只能一次进入一人
    进行计算，线程 t1，t2 想象成两个人
  • 当线程 t1 执行到 synchronized(room) 时就好比 t1 进入了这个房间，并锁住了门拿走了钥匙，在门内执行
    count++ 代码
  • 这时候如果 t2 也运行到了 synchronized(room) 时，它发现门被锁住了，只能在门外等待，发生了上下文切
    换，就阻塞住了，需要等待
  • 这中间即使 t1 的 cpu 时间片不幸用完，被踢出了门外（不要错误理解为锁住了对象就能一直执行下去哦），
    这时门还是锁住的，t1 仍拿着钥匙，t2 线程还在阻塞状态进不来，只有下次轮到 t1 自己再次获得时间片时才
    能开门进入。（也就是上锁后，在执行 count++ 时，释放锁之前，这个时候发生上下文切换，锁依然在 t1 手里，谁也进不来）
  • 当 t1 执行完 synchronized{} 块内的代码，这时候才会从 obj 房间出来并解开门上的锁，唤醒 t2 线程把钥
    匙给他。t2 线程这时才可以进入 obj 房间，锁住了门拿上钥匙，执行它的 count-- 代码

思考

synchronized 实际是用对象锁保证了临界区内代码的原子性，临界区内的代码对外是不可分割的，不会被线程切换所打断。
接下来思考下面的问题

  • 如果把 synchronized(obj) 放在 for 循环的外面，如何理解？-- 原子性
    那么在循环执行前把这个锁拿到，那么就会先执行某个线程的 5000 次 count++ 或者 count-- 操作之后，释放锁，再去执行另外一个线程的 5000 次循环
  • 如果 t1 synchronized(obj1) 而 t2 synchronized(obj2) 会怎样运作？-- 锁对象
    那没什么作用，因为不会出现两个对象抢一把锁的情况，加锁没有意思了，那和不加 synchronized 是一样的
  • 如果 t1 synchronized(obj) 而 t2 没有加会怎么样？如何理解？-- 锁对象
    同第二个，这个时候，t1 线程拿到锁，但是 t2 不需要锁就能执行，那就不需要申请锁了，和 t1 同样冲突，还是和不加 synchronized 一样

哪个线程来访问这个方法都直接进，多个线程可能同时执行该方法的某片代码，所以方法如果存在对共享资源的读写操作，就可能出现并发问题。

变量的线程安全分析

成员变量和静态变量是否线程安全？

  • 如果它们没有共享，则线程安全
  • 如果它们被共享了，根据它们的状态是否能够改变，又分两种情况
    • 如果只有读操作，则线程安全
    • 如果有读写操作，则这段代码是临界区，需要考虑线程安全

如果一个类的一个对象，被多个线程访问，并且有读写操作，那成员变量可能就是不安全的

代码示例
class Counter {
    int count = 0;

    public void add() {
        count++;
    }
}

Counter c = new Counter();
new Thread(c::add).start();
new Thread(c::add).start();

局部变量是否线程安全？

  • 局部变量是线程安全的
  • 但局部变量引用的对象则未必
    • 如果该对象没有逃离方法的作用访问，它是线程安全的
    • 如果该对象逃离方法的作用范围，需要考虑线程安全

局部变量线程安全分析

public static void test1(){
    int i = 10;
    i++;
}

每个线程调用 test1() 方法时局部变量 i，会在每个线程的栈帧内存中被创建多份，因此不存在共享

线程调用方法会在各自的线程栈中生成栈帧，并且各自有各自的成员变量 i

但是如果局部变量引用的是对象就稍微不一样了。

class ThreadUnsafe{
    ArrayList<String> list = new ArrayList<>();
    public void method1(int loopNumber){
        for (int i = 0; i < loopNumber; i++) {
            method2();
            method3();
        }
    }

    private void method2(){
        list.add("1");
    }

    private void method3(){
        list.remove(0);
    }
}

执行

static final int THREAD_NUMBER = 2;
static final int LOOP_NUMBER = 200;
public static void main(String[] args) {
    ThreadUnsafe test = new ThreadUnsafe();
    for (int i = 0; i < THREAD_NUMBER; i++) {
        new Thread(() -> {
            test.method1(LOOP_NUMBER);
        }, "Thread-" + i).start();
    }
}

其中一种情况是 add()，我们简化为执行两步

  • elementData[size] = "1"
  • size++

基本有三种情况

  • 线程 1,2 同时读到 size=0，同时给 elementData[0] 赋值 "1"，最后都赋值 size=1，add 两次但是只有一个值。元素丢失(add与add)
  • 当 list 刚好一个元素，两个线程都来删
    • 一个先删，一个后删后面删的时候前面已经删了，检查不通过，方法没执行，报错 IndexOutOfBoundsException
    • 都检查通过，前面删完为 0，后面删完为 -1，就报错 ArrayIndexOutOfBoundsException(remove与remove 删空了还能删)
  • 初始list=["1"],size=1。
    • T1 开始 add，读到 size=1
    • 这个时候 T2.remove，把 0 号元素删除，赋值 size=0。此时 list=[null, null,...], size=0
    • 之后，T1 赋值并修改 size，list=[null, "1", ...], size=2(add与remove, size与数组内容不一致)

分析：

  • 无论哪个线程中的 method2 和 method3 引用的都是同一个对象中的 list 成员变量

为什么 ArrayList 是线程不安全的呢？因为内部 add() 和 remove() 方法不是一步就完成的，简单来说起码会修改 elementData 数组和 size 大小。这里就不多说了，看看面经。

将 list 修改为局部变量

static final int THREAD_NUMBER = 2;
static final int LOOP_NUMBER = 200;
public static void main(String[] args) {
    ThreadSafe test = new ThreadSafe();
    for (int i = 0; i < THREAD_NUMBER; i++) {
        new Thread(() -> {
            test.method1(LOOP_NUMBER);
        }, "Thread-" + i).start();
    }
}
class ThreadSafe{
    public void method1(int loopNumber){
        ArrayList<String> list = new ArrayList<>();
        for (int i = 0; i < loopNumber; i++) {
            method2(list);
            method3(list);
        }
    }

    private void method2(ArrayList<String> list){
        list.add("1");
    }

    private void method3(ArrayList<String> list){
        list.remove(0);
    }
}

那么就不会有上述问题了

分析：

  • list 是局部变量，每个线程调用时会创建其不同实例，没有共享
  • 而 method2、method3 的参数是从 method1 中传递过来的，与 method1 中引用同一个对象

方法访问修饰符带来的思考，如果把 method2 和 method3 的方法修改为 public 会不会出现线程安全问题？
原来都是 private 的时候，只能通过 method1 来创建 list，每个线程都是独立访问这一个，不会有线程安全问题。
改为 public 后：

  • 情况1：有其它线程调用 method2 和 method3
    也不会有线程安全问题，因为对象是 method1 创建的，在外部拿不到这个引用来传入 method2、method3，也就不会有问题。

  • 情况2：在 情况1 的基础上，为 ThreadSafe 类添加子类，子类覆盖 method2 或 method3 方法，即

    class ThreadSafe{
    public void method1(int loopNumber){
        ArrayList<String> list = new ArrayList<>();
        for (int i = 0; i < loopNumber; i++) {
            method2(list);
            method3(list);
        }
    }

    public void method2(ArrayList<String> list){
        list.add("1");
    }

    public void method3(ArrayList<String> list){
        list.remove(0);
    }
}

class ThreadSafeSubClass extends ThreadSafe{

    @Override
    public void method3(ArrayList<String> list) {
        new Thread(() -> { // T3
            list.remove(0);
        }).start();
    }
}

    这个时候肯定有问题，自己在子类中开了一个新线程访问 list，这个局部变量 list 的引用逃逸了，也就是上面说的逃离了方法的作用范围。这就出现多线程访问同一个变量了

    测试

    static final int THREAD_NUMBER = 2;
static final int LOOP_NUMBER = 200;
public static void main(String[] args) {
    ThreadSafe test = new ThreadSafeSubClass();
    for (int i = 0; i < THREAD_NUMBER; i++) {
        new Thread(() -> { // T1, T2
            test.method1(LOOP_NUMBER);
        }, "Thread-" + i).start();
    }
}

    解析

    但是这次出现问题的地方不再是主线程创建的 T1, T2 之间的冲突，而是 T1(T2 同理) 和它运行时的子线程 T3 的冲突。比如这里，就是主线程调用 method2 进行加，之后调用 method3 进行删除，但是不会等它删除完的，而是只开启一个新线程并且启动，然后返回去执行 add 了，也就是说一个线程在 add，同一个 list 的另一个线程在 remove。

从这个例子可以看出 private 或 final 提供【安全】的意义所在，请体会开闭原则中的【闭】
比如这里我们可以给 method2、method3 都加上 private 修饰符，给 method 加上 final，也就是可以被访问，但是不能被子类重写。

不可变类线程安全性

String、Integer 等都是不可变类，因为其内部的状态不可以改变，因此它们的方法都是线程安全的
String 有 replace，substring 等方法可以改变值啊，那么这些方法是如何保证线程安全的呢？就是新创建一个对象

分析例2 和例 4 得到结论，判断是否是安全的

  1. 先看当前对象是否会被多个线程共享
  • 如果不会共享，通常安全
  • 如果会共享，继续往下看
  1. 看类中是否存在共享的可变成员状态
  • 如果没有，通常安全(比如只有局部变量、只有只读配置、只有不可变对象，并且引用本身不会被并发修改)
  • 如果有，继续分析这些成员变量
  1. 分析成员变量本身
  • 如果是不可变对象，并且引用不被并发修改，通常安全 (final String 引用不会被修改，所以 String 不一定安全，要看具体代码，可能被改引用呢？)
  • 如果是 HashMap、ArrayList 这类非线程安全容器，在共享并发访问时通常不安全 (存在并发读写/写写才通常不安全)
  • 如果是自定义对象，需要继续递归分析该对象是否线程安全 --> (类似 String，先判断这个引用能不能修改，然后看 String 是否安全。不能被修改，且这个类线程安全那就安全了)
  1. 看对这些共享可变状态的访问方式
  • 如果有 synchronized、Lock、原子类、线程安全容器、线程封闭等保护，可能安全
  • 如果没有保护，通常不安全

分析 UserDaoImpl

  • MyServlet 只有一份，会被并发访问，所以 UserService 只有一份，同理 UserDao 也只有一份，UserDaoImpl 就会被并发访问
  • 类中存在可共享的成员变量 conn，这个引用可以被并发修改！线程不安全
    • 线程 A getConnection 之后，线程 B 紧接着也 getConnection，那么 A 继续往下执行的代码是错误的连接了，，，即使同一个数据库，可能权限不同
    • 线程 A 连接后，线程 B 也连接了，但是线程 B 执行快，直接 close() 了，线程 A 傻眼了。(还存在资源泄漏问题，因为这样线程 A 找不到引用了，没有关闭连接，但是也没人用这个连接了)

被多个线程共同访问，并且内容还能被修改的数据

出现金额大于原始账户的情况，肯定是二者余额都够转账才会出现问题，因为一个账户余额不足，相当于这个线程就停了，不用转账了。

给 transfer 加 synchronized 可以吗？

Monitor(锁)

synchronized 底层实现

Monitor 被翻译为监视器或管程。每个 Java 对象都可以关联一个 Monitor 对象，如果使用 synchronized 给对象上锁（重量级）之后，该对象头的 Mark Word 中就被设置指向 Monitor 对象的指针

  • 都可以关联一个 Monitor 对象，不是说一定会关联，没有 synchronized 时是不会有的
  • Monitor 是属于操作系统的，不归Java

利用多个线程在执行下面这个代码的时候

synchronized(obj){
    // 临界区代码
}
  • 刚开始 Monitor 中 Owner 为 null(跟 obj)没有任何关联
  • 当 Thread-2 执行 synchronized(obj) 就会将 Monitor 的所有者 Owner 置为 Thread-2(Monitor 中只能有一个 Owner)
    对象的 Mark Word 也会发生变化，指向 Monitor 的地址，加锁状态 01 代表未加锁，10 代表加锁了
  • 在 Thread-2 上锁的过程中，如果 Thread-3，Thread-4，Thread-5 也来执行 synchronized(obj)，它会先检查这个 obj 有没有关联 Monitor，发现关联了，看看 Owner 有没有人，发现有，就会进入 EntryList BLOCKED，进行等待
  • Thread-2 执行完同步代码块的内容，然后唤醒 EntryList 中等待的线程来竞争锁，竞争的时是非公平的，可能是 Thread-3，可能是 Thread-4，不一定先来先获得，需要根据 JDK 底层实现

  • synchronized 必须是进入同一个对象的 monitor 才有上述的效果
  • 不加 synchronized 的对象不会关联监视器，不遵从以上规则

刚开始没有其他线程竞争时，线程 1 多次访问这个类的多个对象。对象在第一次被线程 1 加锁后，会偏向线程 1，后续线程 1 再进入同步块时，基本只需要检查对象头里的偏向信息，成本很低。

如果线程 2 偶尔访问某些已经偏向线程 1 的对象，就会触发这些对象的偏向撤销。频率低时，JVM 认为这只是个别对象的偶发现象，偏向锁策略整体仍然划算。

如果同一个类的对象频繁发生偏向撤销，达到批量重偏向阈值后，JVM 会认为这个类的对象可能换了主要访问线程，于是对这个类进行批量重偏向。可以粗略理解为：原来那批“刻着线程 1 名字”的标记过期了，后续再被线程 2 使用时，可以重新偏向到线程 2。

如果之后这个类的对象仍然频繁发生偏向撤销，达到批量撤销阈值后，JVM 会认为这个类不适合使用偏向锁，于是禁用该类对象的偏向能力，后续直接走轻量级锁等路径。

synchronized 原理进阶

轻量级锁

Monitor 是重量级锁，不要和本节搞错

轻量级锁的使用场景：如果一个对象虽然有多线程访问，但多线程访问的时间是错开的（也就是没有竞争，如果有竞争还是会升级为重量级锁），那么可以使用轻量级锁来优化。

轻量级锁对使用者是透明的，即语法仍然是 synchronized

假设有两个方法同步块，利用同一个对象加锁，method1 内部会调用 method2

static final Object obj = new Object();
public static void method1() {
    synchronized( obj ) {
        // 同步块 A
        method2();
    }
}
public static void method2() {
    synchronized( obj ) {
        // 同步块 B
    }
}

当执行 synchronized(obj) 时，线程在自己的栈帧里操作如下

  • 创建锁记录（Lock Record）对象，每个线程的栈帧都会包含一个锁记录的结构，内部可以存储锁定对象的 Mark Word

  • 让锁记录中 Object reference 指向锁对象，并尝试用 cas 替换 Object 的 Mark Word，将 Mark Word 的值存入锁记录

    • 01 表示无锁状态，00 表示轻量级锁

    CAS 是 compare and swap，它是CPU提供的原子操作，不会被线程切换打断，意思是如果 obj 的 MarkWord 还是原来的 Mark Word 就把它替换为 Lock Record 的地址 + 00

  • 如果 cas 替换成功，对象头中存储了锁记录地址和状态00，表示由该线程给对象加锁，这时图示如下

  • 如果 cas 失败，有两种情况

    • 如果是其它线程已经持有了该 Object 的轻量级锁，这时表明有竞争，进入锁膨胀过程
    • 如果是自己执行了 synchronized 锁重入，那么再添加一条 Lock Record 作为重入的计数

  • 当退出 synchronized 代码块（解锁时）如果有取值为 null 的锁记录，表示有重入，这时重置锁记录，表示重入计数减一

  • 当退出 synchronized 代码块（解锁时）锁记录的值不为 null，这时使用 cas 将 Mark Word 的值恢复给对象头

    • 成功，则解锁成功
    • 失败，说明轻量级锁进行了锁膨胀或已经升级为重量级锁，进入重量级锁解锁流程
详细流程

  1. 当线程执行到 synchronized (obj) { 底层对应的字节码是 monitorenter

    HotSpot 为了优化 synchronized，不会一上来就用重量级 Monitor，而是先尝试使用轻量级锁。于是当前线程会在自己的栈帧里准备一个结构 Lock Record，注意这个不是对象，不在堆里，而是当前线程栈帧中的一个运行时结构，大致有两个关键信息

    Lock Record 1
├── displaced Mark Word = 暂时为空
└── object reference = obj

    此时，obj 还没被加锁，它的对象头结构是

    [ hashcode | age | biased_lock | 01 ] # 01 表示无锁状态

  2. JVM 会先把这个原始 Mark Word复制到当前线程的 Lock Record 里(这样用来释放锁之后把对象头恢复到原来的样子)，变成

    Lock Record 1
├── displaced Mark Word = obj 原来的 Mark Word
└── object reference = obj

  3. JVM 尝试 CAS 修改 obj 的 Mark Word，如果现在 obj 的 MarkWord 是我刚刚看到的 Mark Word，我就把 obj 的 Mark Word 改成 Lock Record 1 的地址 + 00，如果已经变了，说明有别的线程改过它，CAS 失败。如果 CAS 成功，obj 的 Mark Word 改为了 Lock Record 的地址，说明轻量级锁加锁成功

CAS 失败的情况下，且是自己当前线程执行的 synchronized 锁重入，详细说一下流程

  1. 进入 method2 执行 synchronized (obj) {，创建 method2 栈帧，在这个栈帧里面又会创建一个锁记录

    Lock Record 2
├── displaced Mark Word = 暂时为空
└── object reference = obj

  2. 尝试加锁，发现 obj 已经是轻量级锁，然后JVM判断这个 Lock Record 1 是在自己的栈里面吗？发现是，那就是锁重入，意思是不是别人持有锁，而是我自己在持有锁，所以这个记录是一次重入记录，它不负责真正恢复对象头。

    这个第二个锁是在 method2 栈帧里面创建的，锁记录不是属于栈的，而是栈帧

  3. method2 退出后(底层字节码是 moniterexit)，要释放锁的时候不需要恢复对象头，因为它是一个重入记录，等 method1 执行完，再恢复对象头

锁重入：同一个线程再次获取自己已经持有的锁(看到这个锁的 MarkWord 指向的是自己线程的锁)，那就会再创建一个 Lock Record，只不过 displaced Mark Word 为 null

锁膨胀

如果在尝试加轻量级锁的过程中，CAS 操作无法成功，这时一种情况就是有其它线程为此对象加上了轻量级锁（有竞争），这时需要进行锁膨胀，将轻量级锁变为重量级锁。

static Object obj = new Object();
public static void method1() {
    synchronized( obj ) {
        // 同步块
    }
}
  • 当 Thread-1 进行轻量级加锁时，Thread-0 已经对该对象加了轻量级锁

    Thread-1 加锁的时候，CAS 想用自己的轻量锁地址换 obj 的 Mark Word 时失败，并且这个锁地址也不是我这个线程的
  • 这时 Thread-1 加轻量级锁失败，进入锁膨胀流程
    • 即为 Object 对象申请 Monitor 锁
    • 原来 Thread-0 里面保存的 displaced Mark Word 放到 Monitor 的 header 里
    • 让 Object 的 Mark Word 指向重量级锁地址，锁类型为 10
    • 然后自己进入 Monitor 的 EntryList BLOCKED
  • 当 Thread-0 退出同步块解锁时，使用 cas 将 Mark Word 的值恢复给对象头，但是失败了(CAS 只知道成功或失败，不知道锁类型)，JVM在CAS失败后又读取 obj.Mark Word，看到锁标识是重量级锁。这时会进入重量级解锁流程，即按照 Monitor 地址找到 Monitor 对象，设置 Owner 为 null，唤醒 EntryList 中 BLOCKED 线程，被唤醒的线程自己把自己设置为 Owner。

这个 CAS 怎么就判断到想换的时候是成功还是失败？因为原来的 Mark Word 信息它不是没有吗？中间被人改了它怎么知道？

自旋优化

前面重量级锁竞争里面，如果中间有一个线程也来锁这个对象，那么会进入阻塞队列，但是如果进入阻塞队列，就会发生上下文切换让别的线程来跑任务，然后后面还要再上下文切换回来，这个过程比较消耗时间(因为涉及线程挂起、唤醒、用户态/内核态切换、上下文切换、调度器调度)，这比CPU指令贵很多，所以如果同步块很短，还没切换完呢，人家就释放锁了，还不如让人等一会呢，所以用自旋来优化，也就是不立刻进入阻塞状态，先轮询看看锁有没有释放。

如果当前线程自旋成功（即这时候持有锁线程已经退出了同步块，释放了锁），这时当前线程就可以避免阻塞。

如果自旋重试多次没成功，就进入阻塞队列了。

  • 在 Java 6 之后自旋锁是自适应的，比如对象刚刚的一次自旋操作成功过，那么认为这次自旋成功的可能性会高，就多自旋几次；反之，就少自旋甚至不自旋，总之，比较智能。
  • 自旋会占用 CPU 时间，单核 CPU 自旋就是浪费，多核 CPU 自旋才能发挥优势。
    • 单核 CPU，一次只能跑一个线程，你在跑的时候，拿着锁的线程又动不了，锁肯定释放不了，你转圈没有用
    • 多核 CPU 下，自旋也可能浪费，如果临界区很长，那你转几下获得锁的概率很小，还不如赶紧阻塞，让别的线程跑跑代码
  • Java 7 之后不能控制是否开启自旋功能。自旋是否发生、自旋多少次，由 JVM 根据运行时情况自适应决定。

偏向锁

轻量级锁在没有竞争时（就自己这个线程），每次重入仍然需要执行 CAS 操作。

Java 6 中引入了偏向锁来做进一步优化：只有第一次使用 CAS 将线程 ID 设置到对象的 Mark Word 头之后，发现这个线程 ID 是自己的就表示没有竞争，不用重新 CAS(性能还是有提升的)。以后只要不发生竞争，这个对象就归该线程所有。

偏向状态

Normal 和 Biased 区分是在倒数第三位上

一个对象创建时：

  • 如果开启了偏向锁（默认开启），那么对象创建后，markword 值为 0x05 即最后 3 位为 101，这时它的 thread、epoch、age 都为 0

  • 偏向锁是默认是延迟的，不会在程序启动时立即生效，如果想避免延迟，可以加 VM 参数 -XX:BiasedLockingStartupDelay=0 来禁用延迟

  • 如果没有开启偏向锁，那么对象创建后，markword 值为 0x01 即最后 3 位为 001，这时它的 hashcode、age 都为 0，第一次用到 hashcode 时才会赋值

我们还是开启偏向锁，并且偏向锁启动延迟设置为 0，再跑下面的代码，在打印对象头信息之前调用一下对象的 hashCode() 方法

注意看对象状态变化 01 -> 00 -> 01 也就是 Normal -> Lightweight Locked -> Normal，也就是调用了这个对象的 hashCode() 方法把对象的偏向锁禁用了。

其实也很好理解，因为轻量锁是有锁记录存储原来的对象头，重量级锁有 Monitor 存储原来的对象头，但是偏向锁是直接改动的对象头，也就是把原来存放哈希值的地方存放了线程ID，所以调用了 hashCode 之后再改为偏向锁就不行了，没地放了。所以两个就冲突了，只能选择一个，所以和 前面没调用 hashCode 相比 前面的是位于偏向锁状态，这个一直位于普通状态。

wait notify

为什么需要 wait

  • 由于条件不满足，小南不能继续执行计算

  • 但是小南如果一直占用着锁，等待计算资源的到来，其他人就会一直被阻塞。需要一杯☕️才能继续干活

  • 于是老王单开了一间休息室（调用 wait 方法），让小南到休息室（WaitSet）等着去了，并且此时的锁被释放开，其他人可以由老王随机安排进屋

  • 直到小M将烟送来，大喊一声：你的烟到了（调用 notify ）方法

  • 于是小南就离开了休息室，重新进入竞争锁的队列

原理之 wait/notify

  • Owner 线程发现条件不满足，调用 wait 方法，即可进入 WaitSet 变为 WAITING 状态
  • BLOCKED 和 WAITING 线程都处于阻塞状态，不占用 CPU 时间片
  • BLOCKED 线程会在 Owner 线程释放锁时唤醒
  • WAITING 线程会在 Owner 线程调用 notify 或 notifyAll 时唤醒，但唤醒后并不意味着立刻获得锁，仍需进入 EntryList 重新竞争

API介绍

  • obj.wait()：让进入 object 监视器的线程到 waitSet 等待
  • obj.notify()：让 object 上正在 waitSet 等待的线程中挑一个唤醒
  • obj.notifyAll()：让 object 上正在 waitSet 等待的线程全部唤醒

它们都是线程之间进行协作的手段，都属于 Object 对象的方法，必须获得此对象的锁，成为 Owner 才能调用这几个方法。

wait notify 的正确姿势

首先来看 sleep(long n) 和 wait(long n) 的区别

  • sleep 是 Thread 的静态方法，而 wait 是 Object 的方法，所有对象都有
  • sleep 不强制和 synchronized 配合使用，而 wait 需要和 synchronized 一起用
  • sleep 在睡眠的同时，不会释放对象锁，而 wait 在等待的时候，会释放对象锁

他们的共同点是线程都会进入 TIMED_WAITING 状态

static final Object lock = new Object();
public static void main(String[] args) throws InterruptedException {
    new Thread(() -> {
        synchronized (lock) {
            log.debug("获得锁");
            try {
                Thread.sleep(2000);
                // lock.wait(2000);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }, "t1").start();

    Thread.sleep(1000);
    synchronized (lock) {
        log.debug("获得锁");
    }
}

比如这段代码，让子线程睡2秒，主线程睡1秒，如果是 Thread.sleep(2000)，主线程在这个期间是拿不到锁的，它就会一直等待子线程睡醒，然后才能获得锁，继续向下执行，可以看到主线程在2s后才拿到锁

但如果是 lock.wait()，主线程 1s 后就能拿到锁，因为 lock.wait() 虽然也是睡眠，但是它会释放锁，主线程可以拿到。

这里定义锁用的是 final，这样做的好处是，因为我们加锁都是加在对象上，也就是 synchronized(obj)，如果某些地方不小心修改了 obj，那么加锁就出现了问题，所以为了安全，加上 final，这样 lock 这个引用值就不会变了。

* 同步模式之保护性暂停(设计模式)

定义

即 Guarded Suspension，用在一个线程等待另一个线程的执行结果
要点

  • 有一个结果需要从一个线程传递到另一个线程，让他们关联同一个 GuardedObject，比如这里需要 t1 等待 t2 线程的结果
  • 如果有结果不断从一个线程到另一个线程，那么可以使用消息队列(见生产者/消费者)
  • JDK 中，join 的实现，Future 的实现，采用的就是此模式
  • 因为要等待另一方的结果，因此归类到同步模式

实现

class GuardedObject {
    // 结果
    private Object response;

    // 获取结果的方法
    public Object get() {
        synchronized (this) {
            // 还没有结果
            while (response == null) { // while 解决虚假唤醒
                try {
                    this.wait();
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
            }
            return response;
        }
    }

    // 产生结果
    public void complete(Object response) {
        synchronized (this) {
            // 给结果成员变量赋值
            this.response = response;
            this.notifyAll();
        }
    }
}

增强-超时版

原来那个 t1 线程是会一直等待结果的，也就是无限等待，改写后代码如下

class GuardedObject {
    // 结果
    private Object response;

    // 获取结果的方法
    public Object get(long timeout) {
        synchronized (this) {
            // 开始时间
            long begin = System.currentTimeMillis();
            // 经历时间
            long passedTime = 0;
            // 还没有结果
            while (response == null) { // while 解决虚假唤醒
                // 这一轮应该等待的时间
                long waitTime = timeout - passedTime;
                if (waitTime <= 0) {
                    break;
                }
                try {
                    this.wait(waitTime);
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
                // 经历时间
                passedTime = System.currentTimeMillis() - begin;
            }
            return response;
        }
    }

    // 产生结果
    public void complete(Object response) {
        synchronized (this) {
            // 给结果成员变量赋值
            this.response = response;
            this.notifyAll();
        }
    }
}

三种情况输出分别如下

  • t1 要 get(2000)，t2 1s 就能下载完成，那么 t1 只等一秒就能获取结果
  • t1 要 get(500)，t2 1s 才能下载完成，那么 t1 只能用 null 作为结果返回
  • t1 要 get(2000)，t2 1s 后调用 complete(null)，模拟虚假唤醒一下 t1(类似于其他线程 t3 调用 notifyAll()，t2 还在运行，没获取到结果，response 还是 null)，这个时候因为 get 方法里面 while 条件是 response == null，所以最后还是会等 2s，只有真正往 response 里面赋值的线程 t2 去唤醒才有用。

也可以直接把 synchronized 都加到方法上，更简洁。

* 原理之 join

join 是等待另一个线程的结束，保护性暂停是等待另一个线程的结果

实现原理就是基于刚刚的保护性暂停模式

public final synchronized void join(long millis) throws InterruptedException {
    long base = System.currentTimeMillis();
    long now = 0;

    if (millis < 0) {
        throw new IllegalArgumentException("timeout value is negative");
    }

    if (millis == 0) {
        while (isAlive()) { // 判断当前线程是否还存活
            wait(0);
        }
    } else {
        while (isAlive()) {
            long delay = millis - now;
            if (delay <= 0) {
                break;
            }
            wait(delay);
            now = System.currentTimeMillis() - base;
        }
    }
}

比如你有一个 t1 线程，一个 main 线程，你想让 main 线程等待 t1 执行完，你调用的其实是

public static void main(String[] args){
    // ...
    t1.join();
    // ...
}

你看 join 方法有 synchronized 修饰，也就是会对 t1 这个对象加锁，因为是 main 线程在调用，所以 main 线程会进入 t1 对象的 WaitSet。注意 t1 也是一个 Thread 对象。里面的 wait(0) 是 t1.wait()，是让执行这行代码的线程进入 t1 的 WaitSet，也就是 main 线程在等！

main 线程执行 t1.join() // 创建一个 join 方法的栈帧放入 main 线程栈
    ↓
main 线程进入 join 方法
    ↓
join 方法的 this 是 t1 对象
    ↓
因为 join 是 synchronized，所以 main 线程获得 t1 对象锁
    ↓
main 线程发现 t1.isAlive() == true
    ↓
main 线程调用 t1.wait(0)
    ↓
main 线程释放 t1 对象锁
    ↓
main 线程进入 t1 对象的 WaitSet 等待

t1 线程执行结束时，JVM 会在 t1 对应的 Thread 对象上执行类似 notifyAll() 的操作，唤醒等待在这个 Thread 对象上的线程。

解耦【结果等待者】和【结果生产者】

原来的 GuardedObject 有一个不方便的地方：t1 等结果、t2 产生结果这个模型是合理的，但是要求 t1 和 t2 都必须拿到同一个 GuardedObject 对象引用。
如果只有一个任务，这样写问题不大；但如果有多个任务，比如邮递员 A 给张三送信，邮递员 B 给李四送信，邮递员 C 给王五送信，就需要多个 GuardedObject。此时如果还靠手动传递对象引用，就比较麻烦，也容易混乱。所以可以引入一个中间管理类 Mailboxes。它负责创建并管理多个 GuardedObject，每个 GuardedObject 都有一个唯一 id，相当于一个带编号的信箱。居民创建信箱后，根据自己的 GuardedObject 等待结果；邮递员只需要知道信箱 id，就可以从 Mailboxes 中找到对应的 GuardedObject，并调用 complete() 投递结果。

异步模式之生产者/消费者

定义

  • 与前面的保护性暂停中的 GuardedObject 不同，不需要产生结果和消费结果的线程一对一
  • 消费队列可以用来平衡生产和消费的线程资源
  • 生产者仅负责产生结果数据，不关心数据该如何处理，而消费者专心处理结果数据
  • 消息队列是有容量限制的，满时不会再加入数据，空时不会再消耗数据
  • JDK 中各种阻塞队列，采用的就是这种模式

实现

MQ 是进程间通信，我们做的是 Java 线程之间的通信，不要混淆

@Slf4j(topic = "c.MessageQueue")
// 消息队列类
class MessageQueue {
    // 消息的队列集合
    private LinkedList<Message> list = new LinkedList<>();
    // 队列容量
    private int capacity;

    public MessageQueue(int capacity) {
        this.capacity = capacity;
    }

    // 获取消息
    public Message take() {
        synchronized (list) {
            // 检查队列是否为空
            while (list.isEmpty()) {
                try {
                    log.debug("队列为空, 消费者线程等待");
                    // 为空就去list的 waitSet 等待
                    list.wait();
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
            }
            // 从队列头部获取消息并返回
            Message message = list.removeFirst();
            log.debug("已消费消息 {}", message);
            // 当队列是满的时候需要唤醒那些要存入消息的线程
            list.notifyAll();
            return message;
        }
    }

    // 存入消息
    public void put(Message message) {
        synchronized (list) {
            // 检查队列是否已满
            while(list.size() == capacity) {
                try {
                    log.debug("队列已满, 生产者线程等待");
                    list.wait();
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
            }
            // 将消息加入队列尾部
            list.addLast(message);
            log.debug("已生产消息 {}", message);
            // 当队列是空又加入消息的时候需要把等待消息的线程唤醒
            list.notifyAll();
        }
    }
}

// 只能 get，不能 set，一定是线程安全的了
@Getter
@AllArgsConstructor
@ToString
final class Message {
    private int id;
    private Object value;
}
  • LinkedList 不是线程安全的，但是 MessageQueue 可能被并发访问，所以要帮证线程安全，list 是共享的，所以可以在这个上面加锁
  • 还需要互相 notify 来让 WaitSet 里面的线程唤醒，比如原来队列空了，消费者拿不到消息就去 WaitSet 等待了，当存入了消息之后，需要把它们唤醒。

能不能把 notifyAll() 换成 notify()？

不能换成 notify()。如果只有一个消费者线程和一个生产者线程，一般没问题。但是当有多个消费者和多个生产者时就不安全。比如某一时刻队列为空，多个消费者先进入 WaitSet 等待消息。然后来了多个生产者，假设队列容量是 1，第一个生产者 A 抢到 list 锁，其他生产者因为没拿到锁进入阻塞队列。生产者 A 放入一条消息并调用 notify() 唤醒了某个消费者。但是被唤醒的消费者不会立刻执行，它只是从 WaitSet 移到阻塞队列里，接下来还要重新竞争 list 锁。此时其他生产者也会参与竞争锁。如果其他生产者先抢到锁，它们会发现队列已经满了，于是执行 wait()，进入 WaitSet。这样一来，WaitSet 里就可能同时存在消费者和生产者。之后，被唤醒的消费者抢到锁，消费掉这条消息，队列变空，然后调用 notify()。这时本来应该唤醒生产者，因为队列已经有空位了。但 notify() 是随机唤醒 WaitSet 中的一个线程，可能唤醒另一个消费者。如果唤醒的是消费者，它醒来后发现队列还是空的，又会重新进入 WaitSet。此时队列是空的，生产者还在 WaitSet 里，但没人唤醒它们，程序就可能卡住。

这时再来生产者，如果抢到锁，是可以直接生产的；但如果没有新的线程进来，已有的生产者就只能等下一次 notify。而下一次 notify 仍然是随机的，所以多个生产者和多个消费者共用一个 WaitSet 时，应该用 notifyAll()。

notifyAll() 就很好吗？

也没有，正如上面的，多个消费者在 WaitSet，生产者在阻塞队列里面，notifyAll() 也只是把消费者都放到阻塞队列里面，如果是进来的生产者，它还是会进入 WaitSet，并不是说有消息下一次一定来的就是消费者。但是不会出现程序卡住的情况了

关键区别在于：消费者们已经不在 WaitSet 里沉睡了，它们也在阻塞队列里竞争锁，所以迟早会有消费者拿到锁，消费掉消息

特点
与 Object 的 wait & notify 对比

  • wait、notify、notifyAll 必须配合 Object Monitor 一起使用，而 park、unpark 不必
  • park & unpark 是以线程为单位来阻塞和唤醒线程的，而 notify 只能随机唤醒一个等待中的线程，notifyAll 是唤醒所有等待线程，不精确
  • park & unpark 可以先 unpark，而 wait & notify 只能不能先 notify(你先 notify 再 wait，等待的这个线程也不会被唤醒)

* 原理之 park & unpark

每个线程都有自己的一个 Parker 对象(C代码实现，Java看不到)，由三部分组成 _counter、_cond 和 _mutex，打个比喻

  • 线程就像一个旅人，Parker 就像他随身携带的背包，_cond 就好比背包中的帐篷，_counter 就好比背包中的备用干粮(0 为耗尽，1 为充足)
  • 调用 park 就是要看需不需要停下来休息
    • 如果备用干粮耗尽，那么就钻进帐篷休息(干粮还是 0)
    • 如果备用干粮充足，那么就不需要停留，继续前进(消耗一份干粮，_counter 变为 0)
  • 调用 unpark，就好比补充干粮，让干粮充足
    • 如果此时还在帐篷，那就唤醒他继续前进(干粮直接吃了，不会补充上)
    • 如果此时线程还在运行(干粮 _counter 变为 1)，那么他下次调用 park 的时候，仅消耗掉备用干粮，不需要停留继续前进
      • 但是由于背包有限，多次调用 unpark 仅会补充一份备用干粮

  1. 当线程调用 Unsafe.park() 方法
  2. 检查 _counter，此时 _counter = 0，获得 _mutex 互斥锁
  3. 线程进入 _cond 条件变量阻塞
  4. 设置 _counter = 0

  1. 调用 Unsafe.unpark(Thread_0) 方法，设置 _counter 为 1
  2. 唤醒 _cond 条件变量中的 Thread_0
  3. Thread_0 恢复运行
  4. 设置 _counter 为 0

先调用 unpark 再调用 park 情况如下

  1. 当线程调用 Unsafe.unpark(Thread_0) 方法，设置 _counter = 1
  2. 当前线程调用 Unsafe.park() 方法
  3. 检查 _counter，此时 _counter = 1，线程无需阻塞，继续执行
  4. 设置 _counter = 0

重新理解线程状态转换

  1. NEW -> RUNNABLE
    当调用 t.start() 方法时，由 NEW -> RUNNABLE

  2. RUNNABLE <-> WAITING
    t 线程使用 synchronized(obj) 获取对象锁之后
    调用 obj.wait() 方法时，t 线程进入 WaitSet，由 RUNNABLE -> WAITING
    调用 obj.notify()、obj.notifyAll()、t.interrupt() 时，进入 EntryList，和别的线程一块竞争锁
    - 竞争锁成功，t 线程由 WAITING -> RUNNABLE
    - 竞争锁失败，t 线程由 WAITING -> BLOCKED

    • 当前线程等待超过了 n 毫秒，或 t 线程运行结束，或调用了当前线程的 interrupt() 方法时，当前线程从 TIMED_WAITING -> RUNNABLE

  7. RUNNABLE <-> TIMED_WAITING

    • 当前线程调用了 Thread.sleep(long n)，当前线程从 RUNNABLE -> TIMED_WAITING
    • 当前线程等待超过了 n 毫秒，当前线程从 TIMED_WAITING -> RUNNABLE

  8. RUNNABLE <-> TIMED_WAITING

    • 当前线程调用 LockSupport.parkNanos(long nanos) 或 LockSupport.parkUntil(long millis) 时，当前线程从 RUNNABLE -> TIMED_WAITING
    • 调用 LockSupport.unpark(目标线程) 或调用了线程的 interrupt() 或等待超时，当前线程从 TIMED_WAITING -> RUNNABLE

  9. RUNNABLE <-> BLOCKED

    • t 线程用 synchronized(obj) 获取锁时，如果竞争失败，会从 RUNNABLE -> BLOCKED
    • 持有 obj 锁的线程的同步代码块执行完毕，会唤醒该对象上所有 BLOCKED 的线程重新竞争，如果其中 t 线程竞争成功，从 BLOCKED -> RUNNABLE，其它失败的线程仍然 BLOCKED

  10. RUNNABLE <-> TERMINATED
    当前线程所有代码运行完毕，进入 TERMINATED

多把锁

多把不相干的锁

一间大屋子具有两个功能：睡觉和学习，这二者互不相干(也就是说可以同时干)。现在小南要学习、小女要睡觉，但是如果共用同一间屋子（仅有一个对象锁）的话，那么并发度很低，小南和小女是串行执行的。解决办法是不对大屋子上锁，而是大屋子的多个房间分别上锁（多个对象锁）

@Slf4j(topic = "c.TestMultiLock")
public class TestMultiLock {

    public static void main(String[] args) {
        BigRoom bigRoom = new BigRoom();
        log.debug("开始了");
        new Thread(bigRoom::study, "小南").start();
        new Thread(bigRoom::sleep, "小女").start();
    }
}

@Slf4j(topic = "c.BigRoom")
class BigRoom {

    public synchronized void sleep(){
        SleepUtil.sleep(2000);
        log.debug("sleeping 2 小时");
    }

    public synchronized void study(){
        SleepUtil.sleep(1000);
        log.debug("study 1 小时");
    }
}

从时间上看，一共会花费三个小时

改造代码，对睡觉和学习的屋子分别搞一把锁

@Slf4j(topic = "c.TestMultiLock")
public class TestMultiLock {

    public static void main(String[] args) {
        BigRoom bigRoom = new BigRoom();
        log.debug("开始了");
        new Thread(bigRoom::study, "小南").start();
        new Thread(bigRoom::sleep, "小女").start();
    }
}

@Slf4j(topic = "c.BigRoom")
class BigRoom {

    private final Object studyRoom = new Object();

    private final Object sleepRoom = new Object();

    public void sleep(){
        synchronized (sleepRoom) {
            SleepUtil.sleep(2000);
            log.debug("sleeping 2 小时");
        }
    }

    public void study(){
        synchronized (studyRoom) {
            SleepUtil.sleep(1000);
            log.debug("study 1 小时");
        }
    }
}

最终实际时间只有两个小时

  • 坏处：如果一个线程需要同时获得多把锁，容易发生死锁
