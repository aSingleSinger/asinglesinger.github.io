进程与线程

进程

进程：程序是静止的，进程实体的运行过程就是进程，是系统进行资源分配的基本单位

  • 程序由指令和数据组成，但这些指令要运行，数据要读写，就必须将指令加载至 CPU，数据加载至内存。在指令运行过程中还需要用到磁盘、网络等设备。进程就是用来加载指令、管理内存、管理 IO 的。
  • 当一个程序被运行，从磁盘加载这个程序的代码至内存，这时就开启了一个进程。
  • 进程就可以视为程序的一个实例。大部分程序可以同时运行多个实例进程（例如记事本、画图、浏览器等，比如开启多个记事本，浏览器），也有的程序只能启动一个实例进程（例如网易云音乐、360 安全卫士等）。

进程的特征：并发性、异步性、动态性、独立性、结构性

线程

线程：线程是属于进程的，是一个基本的 CPU 执行单元，是程序执行流的最小单元。线程是进程中的一个实体，是系统独立调度的基本单位，线程本身不拥有系统资源，只拥有一点在运行中必不可少的资源，与同属一个进程的其他线程共享进程所拥有的全部资源

  • 一个进程之内可以分为一到多个线程。

  • 一个线程就是一个指令流，将指令流中的一条条指令以一定的顺序交给 CPU 执行。

  • Java 中，线程作为最小调度单位，进程作为资源分配的最小单位。在 windows 中进程是不活动的，只是作为线程的容器。

关系：一个进程可以包含多个线程，这就是多线程，比如看视频是进程，图画、声音、广告等就是多个线程

线程的作用：使多道程序更好的并发执行，提高资源利用率和系统吞吐量，增强操作系统的并发性能

并发并行

  • 并行（parallel）：是同一时间动手做（doing）多件事情的能力。
    多核CPU下，每个核都可以调度运行线程，这个时候线程可以是并行的。要运行的线程数一般会比CPU核数多，所以也会出现并发（这个时候也会有竞争，比如两个线程都要用同一个资源，那么其中一个就得等待另一个用完）

  • 并发（concurrent）：是同一时间应对（dealing with）多件事情的能力，这些事不一定在同一时刻运行，比如边炒菜边做饭，中间来回交替，是在并发，但不是并行，因为一个人只能在炒菜或者做饭。

    单核 cpu 下，线程实际还是==串行执行==的。操作系统中有一个组件叫做任务调度器，将 cpu 的时间片（windows 下时间片最小约为 15 毫秒）分给不同的线程使用，只是由于 cpu 在线程间（时间片很短）的切换非常快，人类感觉是同时运行的。总结为一句话就是：微观串行，宏观并行，

一般会将这种 线程轮流使用 CPU 的做法称为并发 concurrent

其实单核CPU下，假如有四个线程，如果并发会比串行执行更耗时，因为有上下文切换耗时。

  1. 单核 cpu 下，多线程不能实际提高程序运行效率，只是为了能够在不同的任务之间切换，不同线程轮流使用 cpu，不至于一个线程总占用 cpu，别的线程没法干活
  2. 多核 cpu 可以并行跑多个线程，但能否提高程序运行效率还是要分情况的
    • 有些任务，经过精心设计，将任务拆分，并行执行，当然可以提高程序的运行效率。但不是所有计算任务都能拆分（参考后文的【阿姆达尔定律】）
    • 也不是所有任务都需要拆分，任务的目的如果不同，谈拆分和效率没啥意义
  3. IO 操作不占用 cpu，只是我们一般拷贝文件使用的是【阻塞 IO】，这时相当于线程虽然不用 cpu，但需要一直等待 IO 结束，没能充分利用线程。所以才有后面的【非阻塞 IO】和【异步 IO】优化

对比

线程进程对比：

  • 进程基本上相互独立的，而线程存在于进程内，是进程的一个子集

  • 进程拥有共享的资源，如内存空间等，供其内部的线程共享

  • 进程间通信较为复，同一台计算机的进程通信称为 IPC（Inter-process communication）。不同计算机之间的进程通信，需要通过网络，并遵守共同的协议，例如 HTTP

  • 线程通信相对简单，因为线程之间共享进程内的内存，一个例子是多个线程可以访问同一个共享变量

    Java 中的通信机制：volatile、等待/通知机制、join 方式、InheritableThreadLocal、MappedByteBuffer

  • 线程更轻量，线程上下文切换成本一般上要比进程上下文切换低

应用

同步异步：

  • 需要等待结果返回，才能继续运行就是同步
  • 不需要等待结果返回，就能继续运行就是异步

多线程可以让方法执行变为异步的。比如说读取磁盘文件时，假设读取操作花费了5秒钟，如果没有线程调度机制，这5秒调用者什么也做不了，其代码都得暂停。所以实际开发钟，如果某些操作，比如视频文件需要转换格式等，这些操作比较费事，就需要开一个新线程，避免阻塞主线程。

多线程可以利用多核CPU，如果只是顺序执行，那不就只能用一个CPU。

Java 线程

创建线程

Java 运行的时候默认就有一个主线程了，就是 main，如果想在这个之外再创建线程，得用下面这些方法

Thread

  • start() 方法底层其实是给 CPU 注册当前线程，并且触发 run() 方法执行
  • 线程的启动必须调用 start() 方法，因为如果线程直接调用 run() 方法，相当于变成了调用对象的方法执行，跟线程没关系了
  • 建议线程先创建子线程，主线程的任务放在之后，因为主线程是顺序执行的，可以先执行子线程去跑，然后再跑主线程（main）

Thread 类本身也是实现了 Runnable 接口，Thread 类中持有 Runnable 的属性，执行线程 run 方法底层是调用 Runnable#run：

// 源码
public class Thread implements Runnable {
    private Runnable target;
    
    // 重写的 Runnable 里面的 run
    @Override
    public void run() {
        if (target != null) {
          	// 底层调用的是 Runnable 的 run 方法
            target.run();
        }
    }
}

第一种 Thread 方法其实就是重写了 Thread 里面的 run() 方法。

Runnable 方式的优缺点：

  • 缺点：代码复杂一点。
  • 优点：
    1. Thread 是把线程和任务合并在了一起，Runnable 是把线程和任务分开了
    2. 用 Runnable 更容易与线程池等高级API配合
    3. 用 Runnable 让任务类脱离了 Thread 继承体系，更灵活

Callable

// The Callable interface is similar to Runnable, in that both are designed for classes whose instances are potentially executed by another thread. A Runnable, however, does not return a result and cannot throw a checked exception.
@FunctionalInterface
public interface Callable<V> {

    V call() throws Exception;
}

和 Runnable 类似，但是它有返回值并且能抛出异常

public class ThreadDemo {
    public static void main(String[] args) {
        Callable call = new MyCallable(); // 定义任务执行+返回值
        FutureTask<String> task = new FutureTask<>(call); // 任务
        Thread t = new Thread(task); // 创建线程
        t.start();
        try {
            String s = task.get(); // 获取call方法返回的结果（正常/异常结果）在这里阻塞等待结果
            System.out.println(s);
        }  catch (Exception e) {
            e.printStackTrace();
        }
    }

public class MyCallable implements Callable<String> {
    @Override//重写线程任务类方法
    public String call() throws Exception {
        return Thread.currentThread().getName() + "->" + "Hello World";
    }
}

这里的 FutureTask 可以通过 task.get() 方法来决定什么时候获取返回的结果，只能读取一次哦。FutureTask 可以把线程任务对象包装成一个==未来任务对象==，因为它继承了 Runnable 对象，所以也可以作为 Thread 的构造参数。

public FutureTask(Callable<V> callable)：未来任务对象，在线程执行完后得到线程的执行结果

  • FutureTask 就是 Runnable 对象，因为 Thread 类只能执行 Runnable 实例的任务对象，所以把 Callable 包装成未来任务对象
  • 线程池部分详解了 FutureTask 的源码

public V get()：同步等待 task 执行完毕的结果，如果在线程中获取另一个线程执行结果，==会阻塞等待==，用于线程同步

  • get() 线程会阻塞等待任务执行完成
  • run() 执行完后会把结果设置到 FutureTask 的一个成员变量，get() 线程可以获取到该变量的值
    具体可以看 FutureTask 的 run 方法，它把 Callable.call() 方法执行完的结果存储起来了，存到 outcome 变量里面。

优缺点：

  • 优点：同 Runnable，并且能得到线程执行的结果
  • 缺点：编码复杂

为什么不像 Runnable 那样，直接作为参数传递到 Thread，而是在这里又借助了 FutureTask？

  1. Thread 和 Runnable 在 JDK1.0就有了，Thread 构造方法设计死了一个标准，就是只接收 Runnable 接口，但是 Callable 在 JDK1.5才有，为了向下兼容，，，FutureTask 类被实现来解决这个问题，继承了 Runnable。
  2. Callable 和 Runnable 不同的是它有返回值，所以线程执行完之后，还需要一个地方存放返回值（最核心的原因），所以有 FutureTask，它的 run 方法调用 Callable 的 call 方法并且把返回值存储起来。

线程运行原理

栈与栈帧

Java Virtual Machine Stacks（Java 虚拟机栈）

我们都知道JVM由堆、栈、方法区所组成，其中栈内存是给谁用的呢？其实就是线程，每个线程启动后，虚拟机就会为其分配一块栈内存。

  • 每个栈由多个栈帧（Frame）组成，对应着每次方法调用时所占用的内存
  • 每个线程只能有一个活动栈帧，对应着当前正在执行的那个方法
public class TestFrames {

    public static void main(String[] args) {
        method1(10);
    }

    private static void method1(int x) {
        int y = x + 1;
        Object m = method2();
        System.out.println(m);
    }

    private static Object method2() {
        Object o = new Object();
        return o;
    }
}

main 函数执行，就是一个主线程，会被分配到一块栈内存，刚开始就是 main，他自己也是一个栈帧，当往下执行到 method1 的时候，method1 作为一个新栈帧加入栈内存，当进入 method2 又是一个新的栈帧，跑完删除，这就不多说了。

栈帧图解

比如如上程序，当程序运行时，把这个 TestFrames 类字节码加载到虚拟机中，存放在方法区。Java虚拟机会启动一个 main 的主线程，并且分配一块线程栈。然后main栈帧就加载到这个里面，其中args是一个字符串数组，对象实体放在堆中，args指向那里。当执行到 method1 时，创建 method1 的栈帧，并且返回地址也就是调用 method1 方法的下一条指令地址。这里程序计数器里面放的是当前要执行的二进制指令，这里为了简化方便看，直接放的代码。CPU查看程序计数器的指令进行执行，比如这里程序计数器放的是 new Object()，那么就会去堆中申请内存，放入对象，并且赋值给局部变量 n。

线程上下文切换（Thread Context Switch）

一些原因导致 CPU 不再执行当前线程，转而执行另一个线程

  • 线程的 CPU 时间片用完
  • 垃圾回收（所有工作线程都要暂停，进行垃圾回收）
  • 有更高优先级的线程需要运行
  • 线程自己调用了 sleep、yield、wait、join、park、synchronized、lock 等方法

前三种都是被动的，第四种是主动暂停自己的线程，让其他线程来运行的。

程序计数器（Program Counter Register）：记住下一条 JVM 指令的执行地址，==是线程私有的==，因为每个线程都有自己要执行的程序，所以每个线程都会有一个。

当 Context Switch 发生时，需要由操作系统保存当前线程的状态（PCB 中），并恢复另一个线程的状态

  • 状态包括程序计数器、虚拟机栈中每个栈帧的信息，如局部变量、操作数栈、返回地址等

  • Context Switch 频繁发生会影响性能，所以线程数不是越多越好，当线程数大于核数的时候，必然会进行线程切换，当大于很多的时候，切换频率会很高。

比如这个图，在上下文切换时，CPU 会将当前线程的程序计数器（PC）和寄存器快照保存到线程控制块（TCB）中；而该线程在内存（JVM Stack）里的各级栈帧信息保持不动，待下次切换回该线程时，再通过恢复 PC 指针和寄存器值来延续之前的执行现场。

start 与 run

start 是启动线程，run 是线程启动以后要执行的代码。

:question: 执行 start 方法和执行 run 方法有什么区别呢？

public static void main(String[] args) {

    Thread t1 = new Thread(() -> {
        log.debug("running...");
    }, "t1");

    t1.run(); // run
    log.debug("running...");
}

public static void main(String[] args) {

    Thread t1 = new Thread(() -> {
        log.debug("running...");
    }, "t1");

    t1.start(); // start
    log.debug("running...");
}

对比 run 和 start 方法运行结果，其实第一个用 run 并没有开启线程，你看输出的两个 running... 都是在 main 线程里面，因为这种调用方式相当于调用普通对象的方法，所以还是在主线程里面。

然后我们看一下 start 前后线程的状态

public static void main(String[] args) {

        log.debug("running...");
    }, "t1");

    System.out.println(t1.getState());
    t1.start();
    System.out.println(t1.getState());
}

start 之前线程是新建的状态，还没有被CPU运行，线程启动以后是 RUNNABLE，代表可以被CPU调度执行了。

不能重复调用 start 方法，否则会报错

sleep 与 yield

sleep：

  1. 调用 sleep 会让当前线程从 Running 进入 Timed Waiting 状态（阻塞）

  2. 其它线程可以使用 interrupt 方法打断正在睡眠的线程，这时 sleep 方法会抛出 InterruptedException

  3. 睡眠结束后的线程未必会立刻得到执行，需要抢占 CPU

  4. 建议用 TimeUnit 的 sleep 代替 Thread 的 sleep 来获得更好的可读性

yield：

  1. 调用 yield 会让当前线程从 Running 进入 Runnable 就绪状态，然后调度执行其他线程
  2. 具体的实现依赖于操作系统的任务调度器（比如只有当前一个线程，yield 后没有别的线程了，就还是让当前线程运行）

不同：任务调度器会给就绪状态的线程分配时间片，但是阻塞状态的不会；并且 yield 没有参数，基本上没有等待时间，但是 sleep 会有个时间参数，会有一个等待时间。

sleep

我们看看状态变化，线程阻塞时候的状态标志，也就是刚刚的第一条，调用 sleep 会让当前线程从 Running 进入 Timed Waiting 状态。

public static void main(String[] args) throws InterruptedException {
    Thread t1 =  new Thread("t1"){
        public void run() {
            try {
                Thread.sleep(2000);
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            }
        }
    };

    t1.start();
    log.debug("t1 state: {}", t1.getState()); // RUNNABLE

    Thread.sleep(500);
    log.debug("t1 state: {}", t1.getState()); // TIMED_WAITING
}

Thread.sleep() 在哪个线程被调用就是让哪个线程休眠。

然后我们演示第二条：其它线程可以使用 interrupt 方法打断正在睡眠的线程，这时 sleep 方法会抛出 InterruptedException

public static void main(String[] args) throws InterruptedException {

    Thread t1 = new Thread(() -> {
        log.debug("enter sleep...");
        try {
            Thread.sleep(2000);
        } catch (InterruptedException e) {
            log.debug("wake up...");
            e.printStackTrace(); // 红色字是这里打印输出的信息
        }
    }, "t1");
    t1.start();

    Thread.sleep(1000);
    log.debug("interrupt...");
    t1.interrupt(); // 打断 t1 的睡眠
}

interrupt

打断 sleep、wait、join 的线程

这三种方法会阻塞线程，在打断这三种方法会响应中断并抛出异常，同时JVM还是强制将中断标记位重置为 false。

public static void test1() throws InterruptedException {
    Thread t1 = new Thread(() -> {
        log.debug("sleep...");
        try {
            Thread.sleep(5000); // 睡眠 5 秒
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }, "t1");

    t1.start();
    Thread.sleep(1000); // 主线程睡眠 1 秒，等待线程进入睡眠
    log.debug("interrupt");
    t1.interrupt();
    log.debug("打断标记:{}", t1.isInterrupted());
}

子线程进入睡眠后，主线程对子线程进行打断。但是这里打断标记并没有像预设的那样打断标记为 false 啊

原因是主线程和子线程在竞速，当代码运行到 t1.interrupt() 之后

主线程：调用底层内核方法，把 t1 线程的 TCB 标记位改为 true，下一秒立即向下执行

t1 线程：从睡眠中唤醒，恢复寄存器，抛出异常，抛出异常前一刻，JVM 执行指令将标记位重置为 false

所以主线程速度更快一点。

模式之两阶段终止

Two Phase Termination

在一个线程 T1 中如何优雅终止线程 T2？这里的优雅是指给 T2 一个料理后事的机会

错误思路

  1. 使用线程对象的 stop() 方法停止线程
    stop 方法会真正杀死线程，如果这时线程锁住了共享资源，那么当它被杀死后就再也没有机会释放锁，其他线程将永远无法获取锁
  2. 使用 System.exit(int) 方法停止线程
    目的仅是停止一个线程，但这种做法会让整个程序都停止

两阶段终止模式

比如这个监控记录，正常情况下，判断有没有被打断，没有，就睡眠两秒，然后执行监控记录，意思就是两秒监控一次（因为监控没必要每秒都盯着）；如果被打断了，就料理后事，然后结束循环。如果说在正常执行监控记录的时候被打断了，那就是打断标记被设置为 true 了，然后下一轮循环的时候就可以被判断为打断了然后料理后事结束循环。如果说睡眠的时候被打断了，这个时候会抛出异常，会清空打断标记，这个时候我们在异常处理里面要重新设置打断标记。

代码如下

@Slf4j(topic = "c.Test10")
public class Test10 {
    public static void main(String[] args) throws InterruptedException {
        TwoPhaseTermination tpt = new TwoPhaseTermination();
        tpt.start();

        Thread.sleep(3500); // 睡眠3.5秒后停止监控
        tpt.stop();
    }
}

@Slf4j(topic = "c.TwoPhaseTermination")
class TwoPhaseTermination{
    private Thread monitor;

    // 启动监控线程
    public void start() {
        monitor = new Thread(() -> {
            while (true) {
                Thread current = Thread.currentThread();
                if(current.isInterrupted()){
                    log.debug("料理后事");
                    break;
                }
                try {
                    Thread.sleep(1000);
                    log.debug("执行监控记录");
                } catch (InterruptedException e) {
                    e.printStackTrace();
                    // 重新设置打断标记
                    current.interrupt();
                }
            }
        });
        monitor.start();
    }

    // 停止监控线程
    public void stop() {
        monitor.interrupt();
    }
}

另外注意 isInterrupted() 判断是否打断不会清除打断标记。interrupted() 判断当前线程是否被打断，会清除打断标记。

        log.debug("打断状态: {}", Thread.interrupted()); // 重置打断标记

        LockSupport.park(); // 再来一次 park
        log.debug("unpark...");
    }, "t1");

    t1.start();
    sleep(1000);
    t1.interrupt();

线程状态

五种状态

这是从操作系统层面来描述的

  • 【初始状态】 仅是在语言层面创建了线程对象，还未与操作系统线程关联

  • 【可运行状态】（就绪状态）指该线程已经被创建（与操作系统线程关联），可以由 CPU 调度执行

  • 【运行状态】 指获取了 CPU 时间片运行中的状态

    • 当 CPU 时间片用完，会从 【运行状态】 转换至 【可运行状态】，会导致线程的上下文切换

  • 【阻塞状态】

    • 如果调用了阻塞 API，如 BIO 读写文件，这时该线程实际不会用到 CPU，会导致线程上下文切换，进入 【阻塞状态】

    • 等 BIO 操作完毕，会由操作系统唤醒阻塞的线程，转换至 【可运行状态】

    • 与 【可运行状态】 的区别是，对 【阻塞状态】 的线程来说只要它们一直不唤醒，调度器就一直不会考虑调度它们

  • 【终止状态】 表示线程已经执行完毕，生命周期已经结束，不会再转换为其它状态

六种状态

这个是从 Java API 层面来说的，根据 Thread.State 枚举，分为 6 种状态

  • NEW 线程刚被创建，但是还没有调用 start() 方法

  • RUNNABLE 也就是调用了 start() 方法之后。注意，Java API 层面的 RUNNABLE 状态涵盖了 操作系统 层面的【可运行状态】、【运行状态】和【阻塞状态】（由于 BIO 导致的线程阻塞，在 Java 里无法区分，仍然认为是可运行）

  • BLOCKED, WAITING, TIMED_WAITING 都是 Java API 层面对【阻塞状态】的细分，后面会在状态转换一节详述

  • TERMINATED 当线程代码运行结束

线程状态 导致状态发生条件
NEW（新建） 线程刚被创建，但是并未启动，还没调用 start 方法，只有线程对象，没有线程特征
Runnable（可运行） 线程可以在 Java 虚拟机中运行的状态，可能正在运行自己代码，也可能没有，这取决于操作系统处理器，调用了 t.start() 方法：就绪（经典叫法）
Blocked（阻塞） 当一个线程试图获取一个对象锁，而该对象锁被其他的线程持有，则该线程进入 Blocked 状态；当该线程持有锁时，该线程将变成 Runnable 状态
Waiting（无限等待） 一个线程在等待另一个线程执行一个（唤醒）动作时，该线程进入 Waiting 状态，进入这个状态后不能自动唤醒，必须等待另一个线程调用 notify 或者 notifyAll 方法才能唤醒
Timed Waiting （限期等待） 有几个方法有超时参数，调用将进入 Timed Waiting 状态，这一状态将一直保持到超时期满或者接收到唤醒通知。带有超时参数的常用方法有 Thread.sleep 、Object.wait
Teminated（结束） run 方法正常退出而死亡，或者因为没有捕获的异常终止了 run 方法而死亡

临界区

  • 一个程序运行多个线程本身是没有问题的
  • 问题出在多个线程访问共享资源
    • 多个线程读共享资源其实也没有问题
    • 在多个线程对共享资源读写操作时发生指令交错，就会出现问题
  • 一段代码块内如果存在对共享资源的多线程读写操作，称这段代码块为临界区

例如，下面代码中的临界区

static int counter = 0;
static void increment() 
// 临界区
{ 
 counter++;
}
static void decrement() 
// 临界区
{ 
 counter--;
}

count++ 和 count-- 都包含读+写的操作，所以这一个操作包含了读写，counter 是共享资源，所以单独的 { count++; } { count--; } 是临界区。

竞态条件

多个线程在临界区内执行，由于代码的执行序列不同而导致结果无法预测，称之为发生了竞态条件

synchronized 解决方案

* 应用之互斥

为了避免临界区的竞态条件发生，有多种手段可以达到目的。

  • 阻塞式的解决方案：synchronized，Lock
  • 非阻塞式的解决方案：原子变量

本章使用阻塞式的解决方案：synchronized，来解决上述问题，即俗称的【对象锁】，它采用互斥的方式让同一
时刻至多只有一个线程能持有【对象锁】，其它线程再想获取这个【对象锁】时就会阻塞住。这样就能保证拥有锁
的线程可以安全的执行临界区内的代码，不用担心线程上下文切换

虽然 java 中互斥和同步都可以采用 synchronized 关键字来完成，但它们还是有区别的：

  • 互斥是保证临界区的竞态条件发生，同一时刻只能有一个线程执行临界区代码
  • 同步是由于线程执行的先后、顺序不同、需要一个线程等待其它线程运行到某个点

语法

{
    临界区
}

synchronized 理解

可以做这样的类比：

  • synchronized(对象) 中的对象，可以想象为一个房间（room），有唯一入口（门），房间只能一次进入一人
    进行计算，线程 t1，t2 想象成两个人
  • 当线程 t1 执行到 synchronized(room) 时就好比 t1 进入了这个房间，并锁住了门拿走了钥匙，在门内执行
    count++ 代码
  • 这时候如果 t2 也运行到了 synchronized(room) 时，它发现门被锁住了，只能在门外等待，发生了上下文切
    换，就阻塞住了，需要等待
  • 这中间即使 t1 的 cpu 时间片不幸用完，被踢出了门外（不要错误理解为锁住了对象就能一直执行下去哦），
    这时门还是锁住的，t1 仍拿着钥匙，t2 线程还在阻塞状态进不来，只有下次轮到 t1 自己再次获得时间片时才
    能开门进入。（也就是上锁后，在执行 count++ 时，释放锁之前，这个时候发生上下文切换，锁依然在 t1 手里，谁也进不来）
  • 当 t1 执行完 synchronized{} 块内的代码，这时候才会从 obj 房间出来并解开门上的锁，唤醒 t2 线程把钥
    匙给他。t2 线程这时才可以进入 obj 房间，锁住了门拿上钥匙，执行它的 count-- 代码

思考

synchronized 实际是用对象锁保证了临界区内代码的原子性，临界区内的代码对外是不可分割的，不会被线程切换所打断。
接下来思考下面的问题

  • 如果把 synchronized(obj) 放在 for 循环的外面，如何理解？-- 原子性
    那么在循环执行前把这个锁拿到，那么就会先执行某个线程的 5000 次 count++ 或者 count-- 操作之后，释放锁，再去执行另外一个线程的 5000 次循环
  • 如果 t1 synchronized(obj1) 而 t2 synchronized(obj2) 会怎样运作？-- 锁对象
    那没什么作用，因为不会出现两个对象抢一把锁的情况，加锁没有意思了，那和不加 synchronized 是一样的
  • 如果 t1 synchronized(obj) 而 t2 没有加会怎么样？如何理解？-- 锁对象
    同第二个，这个时候，t1 线程拿到锁，但是 t2 不需要锁就能执行，那就不需要申请锁了，和 t1 同样冲突，还是和不加 synchronized 一样

哪个线程来访问这个方法都直接进，多个线程可能同时执行该方法的某片代码，所以方法如果存在对共享资源的读写操作，就可能出现并发问题。

变量的线程安全分析

成员变量和静态变量是否线程安全？

  • 如果它们没有共享，则线程安全
  • 如果它们被共享了，根据它们的状态是否能够改变，又分两种情况
    • 如果只有读操作，则线程安全
    • 如果有读写操作，则这段代码是临界区，需要考虑线程安全

如果一个类的一个对象，被多个线程访问，并且有读写操作，那成员变量可能就是不安全的

代码示例
class Counter {
    int count = 0;

    public void add() {
        count++;
    }
}

Counter c = new Counter();
new Thread(c::add).start();
new Thread(c::add).start();

局部变量是否线程安全？

  • 局部变量是线程安全的
  • 但局部变量引用的对象则未必
    • 如果该对象没有逃离方法的作用访问，它是线程安全的
    • 如果该对象逃离方法的作用范围，需要考虑线程安全

局部变量线程安全分析

public static void test1(){
    int i = 10;
    i++;
}

每个线程调用 test1() 方法时局部变量 i，会在每个线程的栈帧内存中被创建多份，因此不存在共享

线程调用方法会在各自的线程栈中生成栈帧，并且各自有各自的成员变量 i

但是如果局部变量引用的是对象就稍微不一样了。

class ThreadUnsafe{
    ArrayList<String> list = new ArrayList<>();
    public void method1(int loopNumber){
        for (int i = 0; i < loopNumber; i++) {
            method2();
            method3();
        }
    }

    private void method2(){
        list.add("1");
    }

    private void method3(){
        list.remove(0);
    }
}

执行

static final int THREAD_NUMBER = 2;
static final int LOOP_NUMBER = 200;
public static void main(String[] args) {
    ThreadUnsafe test = new ThreadUnsafe();
    for (int i = 0; i < THREAD_NUMBER; i++) {
        new Thread(() -> {
            test.method1(LOOP_NUMBER);
        }, "Thread-" + i).start();
    }
}

其中一种情况是 add()，我们简化为执行两步

  • elementData[size] = "1"
  • size++

基本有三种情况

  • 线程 1,2 同时读到 size=0，同时给 elementData[0] 赋值 "1"，最后都赋值 size=1，add 两次但是只有一个值。元素丢失(add与add)
  • 当 list 刚好一个元素，两个线程都来删
    • 一个先删，一个后删后面删的时候前面已经删了，检查不通过，方法没执行，报错 IndexOutOfBoundsException
    • 都检查通过，前面删完为 0，后面删完为 -1，就报错 ArrayIndexOutOfBoundsException(remove与remove 删空了还能删)
  • 初始list=["1"],size=1。
    • T1 开始 add，读到 size=1
    • 这个时候 T2.remove，把 0 号元素删除，赋值 size=0。此时 list=[null, null,...], size=0
    • 之后，T1 赋值并修改 size，list=[null, "1", ...], size=2(add与remove, size与数组内容不一致)

分析：

  • 无论哪个线程中的 method2 和 method3 引用的都是同一个对象中的 list 成员变量

为什么 ArrayList 是线程不安全的呢？因为内部 add() 和 remove() 方法不是一步就完成的，简单来说起码会修改 elementData 数组和 size 大小。这里就不多说了，看看面经。

将 list 修改为局部变量

static final int THREAD_NUMBER = 2;
static final int LOOP_NUMBER = 200;
public static void main(String[] args) {
    ThreadSafe test = new ThreadSafe();
    for (int i = 0; i < THREAD_NUMBER; i++) {
        new Thread(() -> {
            test.method1(LOOP_NUMBER);
        }, "Thread-" + i).start();
    }
}
class ThreadSafe{
    public void method1(int loopNumber){
        ArrayList<String> list = new ArrayList<>();
        for (int i = 0; i < loopNumber; i++) {
            method2(list);
            method3(list);
        }
    }

    private void method2(ArrayList<String> list){
        list.add("1");
    }

    private void method3(ArrayList<String> list){
        list.remove(0);
    }
}

那么就不会有上述问题了

分析：

  • list 是局部变量，每个线程调用时会创建其不同实例，没有共享
  • 而 method2、method3 的参数是从 method1 中传递过来的，与 method1 中引用同一个对象

方法访问修饰符带来的思考，如果把 method2 和 method3 的方法修改为 public 会不会出现线程安全问题？
原来都是 private 的时候，只能通过 method1 来创建 list，每个线程都是独立访问这一个，不会有线程安全问题。
改为 public 后：

  • 情况1：有其它线程调用 method2 和 method3
    也不会有线程安全问题，因为对象是 method1 创建的，在外部拿不到这个引用来传入 method2、method3，也就不会有问题。

  • 情况2：在 情况1 的基础上，为 ThreadSafe 类添加子类，子类覆盖 method2 或 method3 方法，即

    class ThreadSafe{
    public void method1(int loopNumber){
        ArrayList<String> list = new ArrayList<>();
        for (int i = 0; i < loopNumber; i++) {
            method2(list);
            method3(list);
        }
    }

    public void method2(ArrayList<String> list){
        list.add("1");
    }

    public void method3(ArrayList<String> list){
        list.remove(0);
    }
}

class ThreadSafeSubClass extends ThreadSafe{

    @Override
    public void method3(ArrayList<String> list) {
        new Thread(() -> { // T3
            list.remove(0);
        }).start();
    }
}

    这个时候肯定有问题，自己在子类中开了一个新线程访问 list，这个局部变量 list 的引用逃逸了，也就是上面说的逃离了方法的作用范围。这就出现多线程访问同一个变量了

    测试

    static final int THREAD_NUMBER = 2;
static final int LOOP_NUMBER = 200;
public static void main(String[] args) {
    ThreadSafe test = new ThreadSafeSubClass();
    for (int i = 0; i < THREAD_NUMBER; i++) {
        new Thread(() -> { // T1, T2
            test.method1(LOOP_NUMBER);
        }, "Thread-" + i).start();
    }
}

    解析

    但是这次出现问题的地方不再是主线程创建的 T1, T2 之间的冲突，而是 T1(T2 同理) 和它运行时的子线程 T3 的冲突。比如这里，就是主线程调用 method2 进行加，之后调用 method3 进行删除，但是不会等它删除完的，而是只开启一个新线程并且启动，然后返回去执行 add 了，也就是说一个线程在 add，同一个 list 的另一个线程在 remove。

从这个例子可以看出 private 或 final 提供【安全】的意义所在，请体会开闭原则中的【闭】
比如这里我们可以给 method2、method3 都加上 private 修饰符，给 method 加上 final，也就是可以被访问，但是不能被子类重写。

不可变类线程安全性

String、Integer 等都是不可变类，因为其内部的状态不可以改变，因此它们的方法都是线程安全的
String 有 replace，substring 等方法可以改变值啊，那么这些方法是如何保证线程安全的呢？就是新创建一个对象

分析例2 和例 4 得到结论，判断是否是安全的

  1. 先看当前对象是否会被多个线程共享
  • 如果不会共享，通常安全
  • 如果会共享，继续往下看
  1. 看类中是否存在共享的可变成员状态
  • 如果没有，通常安全(比如只有局部变量、只有只读配置、只有不可变对象，并且引用本身不会被并发修改)
  • 如果有，继续分析这些成员变量
  1. 分析成员变量本身
  • 如果是不可变对象，并且引用不被并发修改，通常安全 (final String 引用不会被修改，所以 String 不一定安全，要看具体代码，可能被改引用呢？)
  • 如果是 HashMap、ArrayList 这类非线程安全容器，在共享并发访问时通常不安全 (存在并发读写/写写才通常不安全)
  • 如果是自定义对象，需要继续递归分析该对象是否线程安全 --> (类似 String，先判断这个引用能不能修改，然后看 String 是否安全。不能被修改，且这个类线程安全那就安全了)
  1. 看对这些共享可变状态的访问方式
  • 如果有 synchronized、Lock、原子类、线程安全容器、线程封闭等保护，可能安全
  • 如果没有保护，通常不安全

分析 UserDaoImpl

  • MyServlet 只有一份，会被并发访问，所以 UserService 只有一份，同理 UserDao 也只有一份，UserDaoImpl 就会被并发访问
  • 类中存在可共享的成员变量 conn，这个引用可以被并发修改！线程不安全
    • 线程 A getConnection 之后，线程 B 紧接着也 getConnection，那么 A 继续往下执行的代码是错误的连接了，，，即使同一个数据库，可能权限不同
    • 线程 A 连接后，线程 B 也连接了，但是线程 B 执行快，直接 close() 了，线程 A 傻眼了。(还存在资源泄漏问题，因为这样线程 A 找不到引用了，没有关闭连接，但是也没人用这个连接了)

被多个线程共同访问，并且内容还能被修改的数据

卖票练习

@Slf4j(topic = "c.ExerciseSell")
public class ExerciseSell {
    // Random 是线程安全的
    static Random random = new Random();
    public static int randomAmount() {
        return random.nextInt(5) + 1;
    }

    public static void main(String[] args) {
        TicketWindow ticketWindow = new TicketWindow(2000);
        // 保存一下 Thread 用来使用 .join()
        List<Thread> list = new ArrayList<>();
        // 用来存储卖出去多少张票
        List<Integer> sellCount = new Vector<>();
        for (int i = 0; i < 2000; i++) {
            Thread t = new Thread(() -> {
                int count = ticketWindow.sell(randomAmount());
                sellCount.add(count);
            });
            list.add(t);
            t.start();
        }

        // 每个子线程都 join 一下，让主线程等子线程跑完
        list.forEach((t) -> {
            try {
                t.join();
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });

        // 买出去的票求和
        log.debug("selled count:{}",sellCount.stream().mapToInt(c -> c).sum());
        // 剩余票数
        log.debug("remainder count:{}", ticketWindow.getCount());
    }

}

// 售票窗口
class TicketWindow {
    private int count;

    public TicketWindow(int count) {
        this.count = count;
    }

    public int getCount() {
        return count;
    }

    public int sell(int amount) {
        if (this.count >= amount) {
            this.count -= amount;
            return amount;
        } else {
            return 0;
        }
    }
}

出现啥问题了呢？？超卖了，只有 2000 单，但是卖出去 2002 单。原因在哪里？

  • amountList 因为是用的 Vector，是线程安全的，所以不用考虑

  • window.sell(..) 和 amountList.add(..) 两个方法的组合也没问题，因为是针对不同的共享变量，所以组合起来也没有线程安全问题。(不同于 table.get("") + table.put(""))

  • List<Thread> list = new ArrayList<>(); 后出现的 list.add(t) 呢？也没有问题，因为它只被主线程调用，没有被其他线程调用。

  • ticketWindow 多个线程都在访问这个变量，但是变量本身是个引用，没有被修改，它没问题

  • sell(...) 方法，它里面有 count 变量，也就是一个 TicketWindow 对象的 count 被共享了，并且有读写操作！所以有问题。

所以想保护修改很简单，加个 synchronized 来保证就可以了。

  • 同黑马点评中超卖问题，其实如果是线程A和线程B下单数量和小于库存也不会有问题，只在线程A和线程B下单数量和大于库存时，但是又都判断库存足够时，都进行了扣减，出现问题
public synchronized int sell(int amount) {
    if (this.count >= amount) {
        this.count -= amount;
        return amount;
    } else {
        return 0;
    }
}

转账

@Slf4j(topic = "c.ExerciseTransfer")
public class ExerciseTransfer {
    public static void main(String[] args) throws InterruptedException {
        Account a = new Account(1000);
        Account b = new Account(1000);
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 1000; i++) {
                a.transfer(b, randomAmount());
            }
        }, "t1");
        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 1000; i++) {
                b.transfer(a, randomAmount());
            }
        }, "t2");
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        // 查看转账2000次后的总金额
        log.debug("total:{}",(a.getMoney() + b.getMoney()));
    }
    // Random 为线程安全
    static Random random = new Random();
    // 随机 1~100
    public static int randomAmount() {
        return random.nextInt(100) +1;
    }
}
class Account {
    private int money;
    public Account(int money) {
        this.money = money;
    }
    public int getMoney() {
        return money;
    }
    public void setMoney(int money) {
        this.money = money;
    }
    public void transfer(Account target, int amount) {
        if (this.money > amount) {
            this.setMoney(this.getMoney() - amount);
            target.setMoney(target.getMoney() + amount);
        }
    }
}

我们取了两次运行结果，一次总金额大于了原始两个账户的和，一次总金额小于了原始两个账户的和。

我们还是来分析一下这个代码，主线程是创建两个线程，一个线程负责从 a 向 b 转账 1000 次，一个线程负责从 b 向 a 转账 1000 次。
所以如果我只运行线程 t1 或者线程 t2 是不会有问题的，因为我只是循环次数多，一个线程执行循环还是按序执行的，真正出现问题的是两个线程相互转账，同时修改自己和对方的账户余额了。

出现金额大于原始账户的情况，肯定是二者余额都够转账才会出现问题，因为一个账户余额不足，相当于这个线程就停了，不用转账了。

假设 a b 都剩 1000，以 a 向 b 转 200，b 向 a 转 300 为例

给 transfer 加 synchronized 可以吗？

没有用！！！synchronized 保证的是不同线程不会进入同一份代码
因为这里锁的是 this，那在调用 a.transfer(b) 和 b.transfer(a) 的时候锁作用在不同的对象，前者锁的 a，后者锁的 b，我们本来就是用一个线程跑 a.transfer(b)，本线程上不会动 b，也不会多次跑 transfer，所以在这个代码里面加不加效果一样。

解决方法

public void transfer(Account target, int amount) {
    synchronized(Account.class){
        if (this.money > amount) {
            this.setMoney(this.getMoney() - amount);
            target.setMoney(target.getMoney() + amount);
        }
    }
}

确实解决了，但是性能变的很差了，不管多少账户，同一时刻只能有一个账户在转账。后面再说改进
