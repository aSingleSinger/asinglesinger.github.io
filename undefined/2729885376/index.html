S3 客户端配置

S3Client 是客户端，负责用S3兼容协议去和 RustFS 通信。

为什么必须开启 forcePathStyle(true)？
解释两种风格：

  • 路径风格：http://localhost:9000/interview-guide/resume/a.pdf
  • 虚拟主机风格：http://interview-guide.localhost:9000/resume/a.pdf

但是在本地开发、自建MinIO/RustFS的时候，后者根本解析不了，主机是 localhost 它知道是谁，但是 interview-guide.localhost 是啥不知道，解析不到主机在哪，所以只能用路径风格。

.endpointOverride(...) 啥作用？为什么要覆盖？
如果不覆盖 endpoint，SDK 会根据 region 选择 AWS S3 的默认服务地址，然后向 AWS 云上的 S3 对象存储发起请求；要正常访问，通常需要 AWS 账号下的认证信息(AWS账号+IAM凭证或者AccessKey+SecretKey，不过AWS官方推荐前者)，并且会按实际使用量计费。

这里如果按照 AWS，我们不用指定 endpoint，而是根据 region 选择对应的 AWS 服务端点(对应的官方入口地址)，然后进行访问。如果指定了就不去这个区域找了。
但是我们有指定的入口了，region 也还需要，兼容S3协议需要带上一个区域值。

RustFS 和 S3 啥关系？
RustFS、OSS对象存储、MinIO 这些是提供存储能力的系统，S3 API 是很多这类系统愿意兼容的一套接口方式。也就是访问这类对象存储的接口规范，RustFS 实现了这套S3兼容接口，所以我们才能用。

两个file.getSize()什么区别？

这两个是放在HTTP请求头里面的元信息，这个 getSize() 是HTTP协议层面的，让接收方在没接受数据之前，知道有个多大数据的要来，要准备内存、磁盘空间。
RequestBody 里面的 file.getSize() 是告诉 SDK 按照这个长度去读，因为流一个特点是只知道有没有，不知道总共多少数据，那就会一直尝试读，直到发生阻塞或错误，所以告诉SDK有多少数据，读完就算完成了。另外能和前面的进行校验，写了多少，读了多少，是否一致。

另外，其实你可以试试，GetObjectRequest、DeleteObjectRequest、HeadObjectRequest 都不能设置这两个属性，Post... 可以。
因为按照 RestFul 风格，Get 用于获取信息，不是上传数据，Delete 是删除，也不用，Head 也是查询元数据，不需要。Post 也是上传，可以而且需要设置的。就相当于原来在浏览器操作的 Get、Post 这种请求头数据需要我们在这里搞了，因为现在我们在用后端发起原来前端发起的HTTP请求，以前是浏览器自动帮我们填写的Header，帮我们算的 Size 和 Type

大文件分片上传

对于大文件上传，比如5G的视频，上传99%的时候，网络断了，这个时候，就需要重新上传了！！！所以解决办法就是分片上传。

简单来说，需要先把文件切分成多个文件分片，然后再上传这些小的文件分片。
前端发送了所有文件分片之后，服务端再把这些文件分片进行合并，这样就得到一个完整文件。

好处有两点

  1. 断点续传：上传文件中途暂停或失败（比如网络问题、手动暂停）之后，不需要重新上传，只需要上传那些未成功上传的文件分片就可以。所以，分片上传是断点续传的基础。
  2. 多线程上传：可以通过多线程同时对一个文件的多个文件分片进行上传，大大加快文件上传的速度。
大文件上传

对于超过100MB的大文件，建议使用分片上传(Multipart Upload)，代码里面没有，对标 FileStorageService.uploadFile 方法

  • partNumber：必须手动维护这个编号，服务器最后组装时，是完全按照编号顺序来排队的
  • eTag：服务器收到分片后返回的一个校验码。必须把它存进 completedParts 列表

最后完成所有分片上传之后，因为你想，服务器它是不知道你什么时候上传完的，所以需要咱们说啥时候上传完，这个 tag 就有作用了，咱们在完成上传的时候，把 eTag、partNumber 整理成一张表发给服务器，服务器核对发现所有分片都在就合并成一个完整文件，然后就正式出现在存储桶了。

分片上传流程

  1. 检查一下文件的格式、大小等等信息并根据文件信息生成文件的唯一标识（如 SHA-256）。

  2. 将需要上传的文件按照一定的分割规则，分割成相同大小的分片，例如将一个 100MB 的文件切割成相等大小的 5 份，每份 20MB；

  3. 初始化一个分片上传任务，返回本次分片上传的唯一标识（后续可以通过该唯一标识定位该次分片上传任务，可实现手动暂停和开始上传）；

  4. 每个分片在发送前，客户端会计算其哈希值（如 SHA-256），并将这个哈希值与分片一起发送给服务器；

  5. 按照一定的策略（串行或并行）发送各个分片；

  6. 服务器接收到分片后，会重新计算分片的哈希值，并与客户端发送的哈希值进行比对；

  7. 如果哈希值匹配，则认为该分片有效，服务器会存储该分片并等待其他分片的到来；如果哈希值不匹配，则认为该分片无效，说明该文件此次上传之前可能被修改过；

  8. 所有分片发送完成后，服务端会进行分片的合成，以得到原始文件。

  9. 再计算合并后的原始文件的唯一标识，与客户端发送的唯一标识进行对比，一致则说明此次文件上传没问题。

什么是秒传?
秒传就是上传之前判断一下服务器是不是已经上传过该文件，如果上传了，直接返回就可以。我们前面设计的先判断文件是否已经存在就是实现的这个功能。

断点续传 & 秒传

  • 检查文件：先检查文件格式、大小等信息是否符合要求(上传文件不能超过多少MB，什么文件格式不接受等)
  • 生成文件唯一标识
    一般情况下，文件等唯一标识可以通过对文件等大小、名称、最后修改时间等信息通过MD5得到，比如下面链接大规格文件等上传优化就是这么做的，但是MD算法已经不推荐使用，建议使用更安全的哈希算法比如 SHA-2，可以减少哈希碰撞，更加安全。前面这个简历业务，是根据文件内容来生成的唯一标识，使用的是SHA-256。guide 哥说文件内容不变，唯一标识就不变这个体验更好(不管文件名)。综合来说推荐根据内容生成唯一标识，MD5速度快，但是容易出现哈希碰撞，SHA更安全，看需求，如果要求速度至上，就还是MD5。
  • 文件是否上传过：就看 Id 是否存在
