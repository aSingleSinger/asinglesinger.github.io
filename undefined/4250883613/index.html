分组查询

where 和 having 区别

  1. where 是分组之前进行过滤，不满足where条件，不参与分组，而 having 是分组之后对结果进行过滤
  2. where 不能对聚合函数进行判断，但是 having 可以

注意

  1. ==执行顺序==

    1. from 指定查询的表
    2. where 每个数据过滤
    3. group by 对数据分类打包
    4. select 搞一下聚合函数，处理想要展示的列
    5. having 对分组之后对结果进行筛选
    6. order by 排序

  2. having/order by 可以使用别名，因为别名是在 select 中定义的，执行顺序在他们两个之前，where 不行，group by 呢在标准的 SQL 里面是不行的，但是 MySQL 做了扩展，支持它是使用了。所以 group by/having/order by 都可以使用别名。

  3. 分组之后，查询的字段一般为聚合函数和分组字段，查询其他字段无任何意义

约束

约束是作用于表中字段上的规则，用于限制存储在表中的数据。目的是保证数据库中数据的正确、有效性和完整性。

约束 描述 关键字
非空约束 限制该字段的数据不能为null NOT NULL
唯一约束 保证该字段的所有数据都是唯一、不重复的 UNIQUE
主键约束 主键是一行数据的唯一标识，要求非空且唯一 PRIMARY KEY
默认约束 保存数据时，如果未指定该字段的值，则采用默认值 DEFAULT
检查约束（8.0.16版本之后） 保证字段值满足某一个条件 CHECK
外键约束 用来让两张表的数据之间建立连接，保证数据的一致性和完整性 FOREIGN KEY

约束是作用于表中字段上的，可以在创建表/修改表的时候添加约束。

外键

用来让两张表的数据之间建立连接，从而保证数据的一致性和完整性。
比如员工的所属部门ID和部门表。当部门表删除了一个数据，但是员工表未变动，就会出现员工还在，但是部门没有了（查询也会出现错误）。所以需要外键来保证数据一致性和完整性

举例

删除/更新行为

行为 说明
NO ACTION 当在父表中删除/更新对应记录时，首先检查该记录是否有对应外键，如果有则不允许删除/更新。（与 RESTRICT 一致）
RESTRICT 当在父表中删除/更新对应记录时，首先检查该记录是否有对应外键，如果有则不允许删除/更新。（与NO ACTION一致）
CASCADE 当在父表中删除/更新对应记录时，首先检查该记录是否有对应外键，如果有，则也删除/更新外键在子表中的记录。
SET NULL 当在父表中删除对应记录时，首先检查该记录是否有对应外键，如果有则设置子表中该外键值为null（这就要求该外键允许取null）。
SET DEFAULT 父表有变更时，子表将外键列设置成一个默认的值（Innodb不支持）

配置时在后面加上 ON UPDATE、ON DELETE 对应的行为即可。如果不指定，默认行为是 RESTRICT

多表查询

表结构之间的关系分为三种

  • 一对多
    比如一个部门有多个员工，一个员工对应一个部门
    实现：在多的一方建立外键，指向一的一方的主键。
  • 多对多
    比如学生与课程的关系
    实现： 建立第三张中间表，中间表至少包含两个外键，分别关联两方主键
  • 一对一
    比如用户和用户详情的关系
    多用于单表拆分，将一张表的基础字段放在一张表中，其他详情字段放在另一张表中，以提高操作效率
    实现：在任意一方加入外键，关联另外一方的外键，并且设置外键为唯一的（UNIQUE）

子查询

SQL语句中嵌套 SELECT 语句，称为嵌套查询，又称子查询

子查询外部的语句可以是INSERT/UPDATE/DELETE/SELECT的任何一个

根据子查询结果不同，分为：

  • 标量子查询（子查询结果为单个值）
  • 列子查询（子查询结果为一列）
  • 行子查询（子查询结果为一行）
  • 表子查询（子查询结果为多行多列）

根据子查询位置，分为：WHERE之后、FROM之后、SELECT 之后。

标量子查询

子查询返回的结果是单个值（数字、字符串、日期等），最简单的形式，这种子查询成为标量子查询。
常用的操作符：= <> > >= < <=

列子查询

子查询返回的结果是一列（可以是多行），这种子查询称为列子查询
常用的操作符：IN、NOT IN、ANY、SOME、ALL

操作符
IN 在指定的集合范围之内，多选一
NOT IN 不在指定的集合范围之内
ANY 子查询返回列表中，有任意一个满足即可
SOME 与ANY等同，使用SOME的地方都可以使用ANY
ALL 子查询返回列表的所有值都必须满足

行子查询

子查询返回的结果是一行（可以是多列），这种子查询称为行子查询。
常用的操作符：=、<>、IN、NOT IN

  • ==可以使用 (a,b) 的形式来进行条件的判断==

表子查询

子查询返回的结果是多行多列，这种子查询称为表子查询。

事务

事务是一组操作的集合，它是一个不可分割的工作单位，事务会把所有的操作作为一个整体一起向系统提交或撤销操作请求，即这些操作要么同时成功，要么同时失败。

==默认MySQL的事务是自动提交的，也就是当执行一条DML语句（插入、更新、删除），MySQL会立即隐式的提交事务==

事务四大特性

ACID

  • ﻿﻿原子性（Atomicity）：事务是不可分割的最小操作单元，要么全部成功，要么全部失败。
  • ﻿﻿一致性（Consistency）：事务完成时，必须使所有的数据都保持一致状态。
  • ﻿﻿隔离性（Isolation）：数据库系统提供的隔离机制，保证事务在不受外部并发操作影响的独立环境下运行。
  • ﻿﻿持久性（Durability）：事务一旦提交或回滚，它对数据库中的数据的改变就是永久的。

并发事务引发问题

多个并发任务在执行过程中出现的问题，如下三种

问题 描述
脏读 一个事务读到另外一个事务还没有提交的数据。
不可重复读 一个事务先后读取同一条记录，但两次读取的数据不同，称之为不可重复读。
幻读 一个事务按照条件查询数据时，没有对应的数据行，但是在插入数据时，又发现这行数据已经存在，好像出现了“幻影”。

  1. 事务A，执行第一个操作 select
  2. 事务A，执行第二个操作 update
  3. 事务B，执行第一个操作 select（A事务没提交，但是B读了）

  1. 事务A，执行第一个操作 select
  2. 事务B进行 commit，update 了数据
  3. 事务A执行了第二个操作
  4. 事务A执行了第三个操作 select （和第一次读的值不一样了）

  1. 事务A查询id=1的数据
  2. 事务B插入id=1的数据并提交
  3. 事务A插入id=1的数据并提交（报错，主键重复）
  4. 事务A再次查询id=1的数据，但是还是查不到
    原因是因为查询的时候是查询的快照的数据，更新插入的时候是实际的数据。

概述

介绍

索引（index）是帮助MySQL高效获取数据的==数据结构==（有序）。在数据之外，数据库系统还维护着满足特定查找算法的数据结构，这些数据结构以某种方式引用（指向）数据，这样就可以在这些数据结构上实现高级查找算法，这种数据结构就是索引。
==索引是一种数据结构==
如果没有索引，就会去查询全表，效率极低

优缺点

优势 劣势
提高数据检索的效率，降低数据库的IO成本 索引列也是要占用空间的。
通过索引列对数据进行排序，降低数据排序的成本，降低CPU的消耗。 索引大大提高了查询效率，同时却也降低更新表的速度，如对表进行INSERT、UPDATE、DELETE时，效率降低。

Hash

哈希索引就是采用一定的hash算法，将键值换算成新的hash值，映射到对应的槽位上，然后存储在hash表中。
如果两个（或多个）键值，映射到一个相同的槽位上，他们就产生了hash冲突（也称为hash碰撞），可以通过链表来解决。

Hash索引特点

  1. ﻿﻿﻿Hash索引只能用于对等比较（=，in），不支持范围查询（between，>，<，⋯）
  2. ﻿﻿﻿无法利用索引完成排序操作
  3. ﻿﻿﻿查询效率高，通常只需要一次检索就可以了(没有哈希碰撞的时候)，效率通常要高于B+tree索引

前缀索引

当字段类型为字符串（varchar，text等）时，有时候需要索引很长的字符串，这会让索引变得很大，查询时，浪费大量的磁盘IO，影响查询效率。此时可以只将字符串的一部分前缀，建立索引，这样可以大大节约索引空间，从而提高索引效率。

语法如下，(n) 代表用这个字段的前 n 个作为键。

MySQL 在构建这个B+树的时候，就只会提取该列每个值的前 n 个字符，这样每个叶子节点里，存的就是 前 n 个字符 + 主键 ID，这样索引占用的磁盘空间直线下降，每个索引页（默认16K）能容纳更多的索引，树的层级就可能变矮，查询需要的I/O次数就会变少。

我们举例看看查询执行做了什么
假设我们已经对 email 字段建立了长度为 4 对前缀索引 idx_email(email(4))。

MySQL 的执行引擎会做下面几步

  1. 截取：提取要查询的字符串的前 4 个字符，也就是 zhan。
  2. 走索引：在 idx_email 的B+树中快速定位到所有前缀为 zhan 的记录。
  3. 获取主键：拿到这批记录对应的 ID（可能会拿到 zhangsan、zhangwei、zhanli 等多个不同邮箱等主键）。
  4. 强制回表（关键）：拿着这些主键ID，回到聚簇索引中查出完整的整行数据
  5. 最终对比： 在内存中将查出的完整 email 字段与 zhangsan@gmail.com 进行精确比对，丢弃不符合的数据，最后将真正的结果返回给你。

确定前缀索引最佳长度 n 的核心技术指标叫做区分度（Selectivity，也称选择性）
区分度的计算公式非常简单：
$$
\text{Selectivity} = \frac{\text{COUNT(DISTINCT LEFT(column, n))}}{\text{COUNT(*)}} \notag
$$
区分度的取值范围在 0 到 1 之间。值越接近 1，说明这个前缀的重复率越低，索引的过滤效果越好，回表次数越少。

前缀索引也存在致命限制

  • 无法使用覆盖索引（Covering Index）： 即使你写的 SQL 是 SELECT email FROM users WHERE email = 'zhangsan@gmail.com'（只查 email 这一列），MySQL 也必须回表去主键索引里捞出完整的 email。因为它不敢确定索引里的 ‘zhan’ 到底是不是你要的完整数据。
  • 无法用于 ORDER BY 排序： 因为索引里只存了前缀，前缀有序不代表完整的字符串有序。
  • 无法用于 GROUP BY 分组： 同理，前缀相同的记录，完整字符串可能不同，无法直接利用索引完成聚合。

实际业务中很少用前缀索引

单列索引&联合索引

创建联合索引的时候，是如下这种，先用第一个字段 phone 来排序，然后同一个 phone 才用 name 进行排序。所以这也是最左前缀原则。


在业务场景中，如果存在多个查询条件，考虑针对查询字段建立索引时，建议建立联合索引，而非单列索引。

这种又查 phone 又查 name 多的，就建立联合索引，这样也能在结果返回 phone, name 的时候避免回表查询。

所以，在建立索引的时候， 尽量使用联合索引（但是字段太多也不好，二三四个还行，太多不好了）。
常用场景

  1. ==多个 AND 条件==，比如订单表中，经常根据“用户 ID”和“订单状态”来查询， 如果只有单列索引，MySQL 只能先通过 user_id 过滤出一堆记录，再回表查出整行数据来判断 status。而联合索引 (user_id, status) 可以在索引树上一次性定位到同时满足两个条件的主键 ID。
  2. ==覆盖索引==，减少回表查询
  3. ==排序与分组==，利用索引天然的有序性来替代内存排序，比如查询某个分类下的商品，并按价格从低到高排列。SELECT * FROM products WHERE category_id = 1 ORDER BY price ASC;。在 B+ 树中，当 category_id 相同时，price 是绝对有序存储的。MySQL 直接按顺序读取即可，不需要在内存中再执行排序算法。

索引设计原则

  1. ﻿﻿针对于数据量较大，且查询比较频繁的表建立索引。（超过一百万数据的数据库并且查询比较多，如果插入、更新比较多也没必要建立索引）
  2. ﻿﻿针对于常作为查询条件（where）、排序（order by）、分组（group by）操作的字段建立索引。
  3. ﻿﻿尽量选择区分度高的列作为索引，尽量建立唯一索引，区分度越高，使用索引的效率越高。（比如手机号、身份证号、姓名）
  4. ﻿﻿如果是字符串类型的字段，字段的长度较长，可以针对于字段的特点，建立前缀索引。
  5. ﻿﻿尽量使用联合索引，减少单列索引，查询时，联合索引很多时候可以覆盖索引，节省存储空间，避免回表，提高查询效率。(注意最左前缀原则)
  6. ﻿﻿要控制索引的数量，索引并不是多多益善，索引越多，维护索引结构的代价也就越大，会影响增删改的效率。
  7. ﻿﻿如果索引列不能存储NULL值，请在创建表时使用NOT NULL约束它。当优化器知道每列是否包含NULL值时，它可以更好地确定哪个索引最有效地用于查询。

  • count(字段) 没有 not null 约束：InnoDB 引擎会遍历整张表把每一行的字段值都取出来，返回给服务层，服务层判断是否为 null，不为 null，计数累加。 有 not null 约束：InnoDB 引擎会遍历整张表把每一行的字段值都取出来，返回给服务层，直接按行进行累加。

  • ﻿﻿count(1)
    InnoDB 引擎遍历整张表，但不取值。服务层对于返回的每一行，放一个数字 1 进去，直接按行进行累加。

  • count(*)
    InnoDB引擎并不会把全部字段取出来，而是专门做了优化，不取值，服务层直接按行进行累加。

  • 对于B-tree，无论是叶子节点还是非叶子节点，都会保存数据，这样导致一页中存储的键值减少，指针跟着减少，要同样保存大量数据，只能增加树的高度，导致性能降低；（也就是前面我们看了，一页的大小就是1M，一个节点1页，如果说要存放指针，那么一页可以有好多指针，如果存放数据，那一页能放的指针就少了，树的高度会增加）因为你要注意，当数据有百万条时，一个节点不可能只有两三个指针的，所以说一个节点可能有非常多的指针，这个时候放数据就占据了指针的空间了。

元数据锁

MDL（元数据锁）加锁过程是系统自动控制，无需显式使用，在访问一张表的时候会自动加上。MDL锁主要作用是维护==表元数据==（也就是表结构数据）的数据一致性，在表上有活动事务的时候，不可以对元数据进行写入操作。==为了避免DML与DDL冲突，保证读写的正确性==。

DML 是数据操作语言，用于对表的数据增删改
DDL 是数据库定义语句，用来定义或修改数据库结构、表结构

在MySQL5.5中引入了MDL，当对一张表进行增删改查的时候，加MDL读锁（共享）；当对表结构进行变更操作的时候，加MDL写锁（排他）。

此时线程B想对表上表锁，但是你想，表锁是不是和行锁冲突，如果有一行有锁，那表锁就上不了了，所以需要检查有没有行锁，就需要遍历整张表，看有没有行锁，这个时候就太麻烦了，所以引入了意向锁。==当一个事务准备给某一行加锁时，必须先给整张表加一个意向锁，所以意向锁是表级锁，但是表达的是行级意向==。

意向锁有两种

  1. ﻿﻿﻿意向共享锁（IS）：由语句 select... lock in share mode 添加。
  2. ﻿﻿﻿意向排他锁（IX）：由 insert、update、delete、select... for update 添加。

单独介绍一下这两个东西

SELECT…LOCK IN SHARE MODE

MySQL8.0更推荐语法 SELECT… FOR SHARE。
默认情况下， MySQL 的普通 SELECT 查询是不加任何锁的， 但在很多业务场景下，我们需要在读取数据的同时把它锁住，防止其他事务修改它
典型使用场景：确认数据是否存在，并确保在后续操作期间它不被改变。
比如说： 你要在子表插入一条记录，前提是父表必须存在对应的 ID。你可以先 SELECT … LOCK IN SHARE MODE 查一下父表，确认存在后再插入子表。这样能防止在你查询确认后、插入子表前，有其他事务把父表里的那条记录删了。

加上之后，别的事务可以进行普通的 SELECT，也可以加 LOCK IN SHARE MODE，加共享锁。大家都可以读，但是其他事务不能再更新、删除。

SELECT…FOR UPDATE

加上之后，别的事务职能进行普通的 SELECT，不能加共享锁 LOCK IN SHARE MODE，不能 SELECT…FOR UPDATE，更不能修改、删除。
典型使用场景：准备更新数据，并且绝对不允许别人同时修改（防止丢失更新）。

比如说： 电商扣减库存、银行转账。你先 SELECT stock FROM table WHERE id=1 FOR UPDATE 把这件商品的库存锁住，然后在程序里计算 new_stock = stock - 1，最后再 UPDATE 回去。如果你只用普通 SELECT，可能有两个并发请求同时读到库存为 10，然后都更新为 9，导致超卖。

回归正传，两个锁特性如下

  1. 意向共享锁（IS）：与表锁共享锁（read）兼容，与表锁排它锁（write） 互斥。
  2. 意向排他锁（IX）：与表锁共享锁（read）及排它锁（write）都互斥。意向锁之间不会互斥。

你想，意向共享锁是怎么添加的？SELECT…LOCK IN SHARE MODE，也就是说它的作用是保证读不会出错，所以不允许修改，所以表锁共享锁可以加，但是不能加表锁排它锁。

意向排他锁是更新、插入、删除的时候都会加行锁，生成意向排他锁，也就是说在锁期间会在什么时机更改是不知道的，表锁想拿到控制只能等这个行锁解除才可以，所以和表锁读/写都互斥

为什么不会互斥呢？
因为你想，意向锁其实只是个标志 （计数器），如果对某一行上锁了，意向锁就存在了，如果我对另一行也上锁，行锁之间不会互斥，意向锁也会多一个，就也不能互斥

行级锁

行级锁，每次操作锁住对应的行数据。锁定粒度最小，发生锁冲突的概率最低，并发度最高。应用在InnoDB存储引擎中。==还记得吗，行级锁是InnDB存储引擎的一大特性==
InnoDB的数据是基于索引组织的，行锁是通过对索引上的索引项加锁来实现的，而不是对记录加的锁。对于行级锁，主要分为以下三类：

  1. 行锁（Record Lock）：锁定单个行记录的锁，防止其他事务对此行进行update和delete。在RC、RR隔离级别下都支持。（Read committed 和 Repeatable Read）。5。单个数据

  2. 间隙锁（Gap Lock）：锁定索引记录间隙（不含该记录），确保索引记录间隙不变，防止其他事务在这个间隙进行insert，产生幻读。在RR隔离级别下都支持。(3,5)

  3. 临键锁（Next-Key Lock）：行锁和间隙锁组合，同时锁住数据，并锁住数据前面的间隙Gap。在RR隔离级别下支持。(3,5]，左开右闭

InnoDB实现了以下两种类型的行锁：

  1. 共享锁（S）：允许一个事务去读一行，阻止其他事务获得相同数据集的排它锁。

  2. 排它锁（X）：允许获取排它锁的事务更新数据，阻止其他事务获得相同数据集的共享锁和排它锁。


    ==SQL语句加锁==情况如下

默认情况下，InnoDB 在 REPEATABLE READ 事务隔离级别运行，InnoDB 使用 next-key 锁进行搜索和索引扫描，以防止幻读。

  1. 针对唯一索引进行检索时，对已存在的记录进行等值匹配时，将会自动优化为行锁。
  2. InnoDB的行锁是针对于索引加的锁，不通过索引条件检索数据，那么InnoDB将对表中的所有记录加锁，此时就会升级为表锁。

怎么有幻读呢？如下场景
假设你的表 stu 中只有两条记录：id=3 和 id=8。

  1. 事务 A 执行：SELECT * FROM stu WHERE id > 3 FOR UPDATE;
  2. 如果没有 Next-Key 锁，事务 A 只锁住了现有的 id=8。
  3. 事务 B 此时插入一条新记录：INSERT INTO stu (id) VALUES (5);（成功插入，因为 id=5 没人锁）。
  4. 事务 A 再次执行同样的查询，发现多出了 id=5 的数据！—— 这就是“幻读”。
    为了解决这个问题，InnoDB 使用 Next-Key 锁，把 (3, 8] 甚至 (8, +∞) 的所有空间都封死。这样事务 B 想在 id=5 的位置插入时，会被挡在门外。

间隙锁/临键锁

默认情况下，InnoDB在 REPEATABLE READ 事务隔离级别运行，InnoDB使用 next-key 锁进行搜索和索引扫描，以防止幻读（不只要锁住已经存在的数据，还要锁住不存在的数据所在的区间）。

  1. 索引上的等值查询（唯一索引），给不存在的记录加锁时，优化为间隙锁。如果存在就变成行锁了。
  2. 索引上的等值查询（普通索引），向右遍历时最后一个值不满足查询需求时，next-key lock 退化为间隙锁。
  3. 索引上的范围查询（唯一索引）–会访问到不满足条件的第一个值为止。
    注意：==间隙锁唯一目的是防止其他事务插入间隙。间隙锁可以共存，一个事务采用的间隙锁不会阻止另一个事务在同一间隙上采用间隙锁==。

==？== 你可能会好奇，只要对 id=5 的加锁不就行了，因为如果出现幻读，也是别的事务插入了 id=5 的才可能出现幻读，对 (3,8) 都加干嘛？

    说一下 lock_mode 的行锁含义

  • S 后面没有后缀，代表 Next-Key Lock，代表记录锁+前面的间隙锁
  • S, GAP 代表纯粹的 Gap Lock，只锁前面的间隙，不锁记录本身
  • S, REC_NOT_GAP 纯粹的行锁，不锁间隙，只锁本身
    lock_data 含义，在这里，第一个值代表 age，第二个值代表 id。（二级索引值，主键值）

详细来看，第一个 IS 为意向共享锁
第二个是在二级索引上加了临键锁，这里是锁住了 (1, 3]
第三个是需要对主键上锁，就是行锁，锁住了 3
第四个是对二级索引上了间隙锁，(3, 7)
这样覆盖的就全面了，在之前之后插入 age=3 的都不会有问题了。

==?== age 上怎么加了两个锁？为什么加两个？
因为二级索引是 B+ 数，排序规则是 age, id，所以相等的 age，可能之前也存在小于 age 的 id，之后也有大于它的 id，所以之前之后都要锁。实际范围已经是 (1, 7) 了。
继续提问 ==?== 那为什么不直接在 7 上加间隙锁？从 (1,7)。
要知道锁的原理，锁是需要放在实体上的，比如放在 (3, 3)，也就是 age=3,id=3 这个实体上，放的如果是行锁那就代表只锁这一个，放的是间隙锁就代表从这个节点到之前的一个节点中间不能插入数据了，如果放的是临键锁，说明当前和之前的都不能插入了。所以在 (7,7) 上没办法只插入一个锁来实现。

==?== 为什么还会对主键加锁？
行锁，临键锁都会对主键加锁，间隙锁不会。
因为加锁也有防止修改数据的因素，如果别人通过主键修改怎么办？所以要对主键加锁。间隙锁不包含存在的数据，所以不用对主键加锁，也没法加（没有实体）

这里对主键加锁是为了防止对数据修改和删除，对二级索引加锁是为了防止数据插入，因为你看如果想插入 age=3 的，那必须修改二级索引，这个时候就违背了间隙锁或者临键锁，不能插入，如果想修改 age=3 的，就得去聚集索引那个地方改，但是也加锁了，不能改，这样数据就安全了！！！！真厉害。

架构

MySQL5.5版本开始，默认使用InnoDB存储引擎，它擅长事务处理，具有崩溃恢复特性，在日常开发中使用非常广泛。下面是InnoDB架构图，左侧为内存结构，右侧为磁盘结构。

内存架构

  • Buffer Pool：缓冲池是主内存中的一个区域，里面可以缓存磁盘上经常操作的真实数据，在执行增删改查操作时，先操作缓冲池中的数据（若缓冲池没有数据，则从磁盘加载并缓存），然后再以一定频率刷新到磁盘，从而减少磁盘IO，加快处理速度。
    缓冲池以Page页为单位，底层采用链表数据结构管理Page。根据状态，将Page分为三种类型（对应图中三种颜色）：
    • ﻿﻿free page： 空闲page，未被使用。
    • ﻿﻿clean page：被使用page，数据没有被修改过。
    • ﻿﻿dirty page：脏页，被使用page，数据被修改过，页中数据与磁盘的数据产生了不一致。

  • Change Buffer： 更改缓冲区（针对于非唯一二级索引页，唯一索引和聚集索引都不会操作这个），在执行DML语句时，如果这些数据Page没有在Buffer Pool中，不会直接操作磁盘，而会将数据变更存在更改缓冲区 Change Buffer 中，在未来数据被读取时，再将数据合并恢复到Buffer Pool中，再将合并后的数据刷新到磁盘中。
    ==?== Change Buffer的意义是什么？
    与聚集索引不同，二级索引通常是非唯一的，并且以相对随机的顺序插入二级索引（聚集索引通常是按顺序增大的，auto_increment）。同样，删除和更新可能会影响索引树中不相邻的二级索引页，如果每一次都操作磁盘，会造成大量的磁盘IO。有了ChangeBuffer之后，我们可以在缓冲池中进行合并处理，减少磁盘IO。

  • Adaptive Hash Index：自适应hash索引，用于优化对Buffer Pool数据的查询。InnoDB存储引擎会监控对表上各索引页的查询，如果观察到hash索引可以提升速度，则建立hash索引，称之为自适应hash索引。（哈希索引按值查询快，但是不能进行范围检索）
    ==自适应哈希索引，无需人工干预，是系统根据情况自动完成==。
    参数: adaptive_hash_index


    默认系统是开启自适应哈希索引的。

  • Log Buffer：日志缓冲区，用来保存要写入到磁盘中的log日志数据（redo log、undo log），默认大小为 16MB，日志缓冲区的日志会定期刷新到磁盘中。如果需要更新、插入或删除许多行的事务，增加日志缓冲区的大小可以节省磁盘I/O。
    参数：
    innodb_log_buffer_size: 缓冲区大小
    innodb_flush_log_at_trx_commit：日志刷新到磁盘时机

    • 1 日志在每次事务提交时写入并刷新到磁盘
    • 0 每秒将日志写入并刷新到磁盘一次
    • 日志在每次事务提交后写入，并每秒刷新到磁盘一次

如果有一台专门的 MySQL 服务器，一般 80% 的内存都会分配给缓冲区。

磁盘结构

  • System Tablespace：系统表空间是更改缓冲区的存储区域。如果表是在系统表空间而不是每个表文件或通用表空间中创建的，它也可能包含表和索引数据。（在MySQL5.x版本中还包含InnoDB数据字典、undolog等）
    也就是说它存放了更改缓冲区的数据，如果说表不是一个表一个文件，而是一个共享空间，那就是放在这里面的。
    参数: innodb_data_file_path


    默认是存放在 ibdata1 文件里面的

  • File-Per-Table Tablespaces：每个表的文件表空间包含单个InnoDB表的数据和索引，并存储在文件系统上的单个数据文件中。（比如 course.ibd，student.ibd）表名.ibd。
    参数: innodb_file_per_table（默认是开启的，也就是说一个表一个文件，如果关闭，所有的表数据和索引都会放在上面这个系统表空间）

  • General Tablespaces：通用表空间，需要通过 CREATE TABLESPACE 语法创建通用表空间，在创建表时，可以指定该表空间。

    ==？== 这个通用表空间干什么用的呢？
    系统表空间是什么？一个数据库一个，必须创建的
    独立表空间，每个表一个的
    通用表空间是什么？我自己创建一个表空间，我可以指定谁放进去，我也可以指定这个文件的位置（可以放到读写更快的SSD）。比如有些表数据很少，一个表只有几条数据的，我可以把这些表放在一个表空间。并且可以跨数据库存表（不同数据库的表可以放在一个文件夹里面，但是不能跨引擎）

  • Undo Tablespaces：撤销表空间，MySQL实例在初始化时会自动创建两个默认的undo表空间（undo001, undo002，初始大小16M），用于存储undo log日志。

  • Temporary Tablespaces: InnoDB使用会话临时表空间和全局临时表空间。存储用户创建的临时表等数据。

  • Doublewrite Buffer Files： 双写缓冲区，innoDB引擎将数据页从Buffer Pool刷新到磁盘前，先将数据页写入双写缓冲区文件中，便于系统异常时恢复数据。

  • Redo Log：重做日志，是用来实现事务的持久性。该日志文件由两部分组成：重做日志缓冲（redo log buffer）以及重做日志文件（redo log），前者是在内存中，后者在磁盘中。当事务提交之后会把所有修改信息都会存到该日志中，用于在刷新脏页到磁盘时，发生错误时，进行数据恢复使用。
    不会永久保存，每隔一段时间会清理之前的 redo log。

后台线程

用来在合适的时机将缓冲池的数据刷新到磁盘

  1. Master Thread
    核心后台线程，负责调度其他线程，还负责将缓冲池中的数据异步刷新到磁盘中，保持数据的一致性，还包括脏页的刷新、合并插入缓存、undo页的回收。

  2. IO Thread
    在InnoDB存储引擎中大量使用了AIO（异步IO）来处理IO请求，这样可以极大地提高数据库的性能，而IO Thread主要负责这些IO请求的回调。

  3. Purge Thread
    主要用于回收事务已经提交了的undo log，在事务提交之后，undo log可能不用了，就用它来回收。

事务原理


事务的四个特性，原子性、一致性、持久性由 redo log，undo log 文件维护，隔离性由锁、MVCC机制实现

redo log

重做日志，记录的是事务提交时数据页的物理修改，是用来实现事务的持久性。
该日志文件由两部分组成：重做日志缓冲（redo log buffer）以及重做日志文件（redo log file），前者是在内存中，后者在磁盘中。当事务提交之后会把所有修改信息都存到该日志文件中，用于在刷新脏页到磁盘，发生错误时，进行数据恢复使用。


比如说有一个事务，里面有很多操作，在操作时会先去 buffer pool 看有没有该页的数据，如果没有，就会通过后台线程将数据从磁盘读取出来缓存在内存中，修改后缓存区的页就变成了脏页。改完之后，数据页的变化会存储在 Redolog buffer 里面，当事务提交的时候，这个数据页变化信息会刷新到磁盘里面，等后面脏页数据刷新到磁盘的时候，如果出现错误，这个时候就可以根据 ib_logfile0/1 来进行恢复（这个恢复到意思是把事务恢复到事务提交之后的状态，不是之前的，因为事务已经提交了，出错了也不能去撤回了)。当脏页数据刷新到磁盘之后，redo log 磁盘文件记录的也就没用了，所以 ib_logfile0/1 是循环写入的，不会永久保存。

==?== 事务提交的时候为什么不直接刷新到磁盘？
因为通常事务是包含很多条语句，通常是随机读取I/O的，性能低。如果有 Redolog，先把数据页变化异步刷新到磁盘，因为日志文件都是顺序记录的，所以I/O速度较快，这种机制叫WAL。这样过一段时间之后再把脏页数据刷新到磁盘里面。
==?== ib_logfile0/1怎么用的？
顺序写满 0，0写满后，切换到1，1写满了，跳回0开头重新覆盖写。
==?== 那如果刷新很慢怎么办？两个日志文件都满了，但是脏数据还没刷过去
MySQL就会阻塞所有的更新操作，把脏页全部刷到磁盘，腾出Redo Log空间，这也是著名的由于日志写满导致的性能抖动。

undo log

用来保证原子性。

回滚日志，用于记录数据被修改前的信息，作用包含两个：提供回滚和 MVCC（多版本并发控制）。
undo log和redo log记录物理日志不一样，它是逻辑日志（redo log记录的是里面有什么数据，叫物理日志，undo log记录的是什么操作，叫逻辑日志）。可以认为当delete一条记录时，undo log中会记录一条对应的insert记录，反之亦然，当update一条记录时，它记录一条对应相反的update记录。当执行rollback时，就可以从undolog中的逻辑记录读取到相应的内容并进行回滚。

Undolog销毁：undo log在事务执行时产生，事务提交时，并不会立即删除undo log，因为这些日志可能还用于MVCC。
Undo log存储：undo log采用段的方式进行管理和记录，存放在前面介绍的 rollback segment 回滚段中，内部包含1024个undo log segment.

MVCC

当前读
==读取的是记录的最新版本，读取时还要保证其他并发事务不能修改当前记录，会对读取的记录进行加锁==。对于我们日常的操作，如：select...lock in share mode（共享锁），select..for update、update、insert、delete（排他锁）都是一种当前读。
快照读
简单的select（不加锁）就是快照读，快照读，读取的是记录数据的可见版本，有可能是历史数据，不加锁，是非阻塞读。

  • Read Committed：每次select，都生成一个快照读。
  • Repeatable Read：开启事务后第一个select语句才是快照读的地方。
  • Serializable：快照读会退化为当前读。
    MVCC
    全称 Multi-Version Concurrency Control，多版本并发控制。指维护一个数据的多个版本，使得读写操作没有冲突，快照读为MySQL实现MVCC提供了一个非阻塞读功能。MVCC的具体实现，还需要依赖于数据库记录中的三个隐式字段、undo log日志、readView。

undo log

回滚日志，在insert、update、delete的时候产生的便于数据回滚的日志。
当insert的时候，产生的undolog日志只在回滚时需要，在事务提交后，可被立即删除。
而update、delete的时候，产生的undo log日志不仅在回滚时需要，在快照读时也需要，不会立即被删除。
那么何时删除？

  • 事务提交后：
    • 对于INSERT操作，事务提交后，undo log可以被立即删除，因为不再需要用于回滚。
    • 对于UPDATE和DELETE操作，undo log不会立即被删除，因为它们可能在后续的快照读取中被使用。
  • 快照读取结束：
    • 当所有依赖于该undo log的快照读取操作结束后，undo log才会被删除。这意味着如果有一个事务正在进行快照读取，并且依赖于某个undo log，那么这个undo log会一直保留直到该事务结束。

readview

为什么有 readview？
当你执行一个普通的 SELECT 时，Undo Log 版本链上可能堆了一长串不同事务修改过的记录。

  • 有的是已经提交的。
  • 有的是还没提交的。
  • 有的是在你查询之后才开始的。

ReadView（读视图）是快照读 SQL执行时MVCC提取数据的依据，记录并维护系统当前活跃的事务（未提交的）id。
ReadView中包含了四个核心字段：

  • max_trx_id 不是活跃事务最大ID+1，有个事务ID可能更大，但是不活跃了。
  • creator_trx_id 意思是当一个事务进行查询的时候，他会生成一个 readview，哪个事务成的这个 readview 呢？就记录在这个字段里面

图中的 trx_id 代表 undo log 里面的 DB_TRX_ID。
